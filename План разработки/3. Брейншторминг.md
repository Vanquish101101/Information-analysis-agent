# 3. Брейншторминг — Information Analysis Agent (Агент 3)

Дополнительные идеи и решения поверх того, что уже зафиксировано в `1. Идея.md` и
`2. Исследование.md`. Задача этого файла — не повторять решённое, а обсудить, что можно добавить,
и закрыть оставшиеся риски/вопросы перед ТЗ.

## 1. Дополнительные возможности (поверх базовой задачи)

Базовая задача Агента 3 (контроль качества входа, семантический синтез, подготовка выхода для
Агента 4) уже решена. Ниже — идеи, что можно добавить сверху, раз у агента и так есть память
(pgvector) и схема `entities/claims/sources`:

- **Рейтинг источников со временем.** Раз Агент 3 и так сверяет факты между источниками
  (§2.2 исследования), он может попутно вести статистику: какие источники чаще подтверждаются,
  какие чаще противоречат остальным — и повышать/понижать вес источника в будущих сверках.
  Это прямое развитие идеи "Рейтинг источников" из брейншторма Агента 1 (§1), но реализуемо
  только здесь, потому что только Агент 3 видит кросс-источниковую картину.
- **Отслеживание изменений нарратива во времени.** Поскольку факты хранятся в `claims` с
  провенансом, можно сравнивать "что говорили об X неделю назад vs сейчас" — не просто снимок,
  а динамика. Прямое развитие идеи Агента 1 "Сравнение периодов", но не по постам, а по фактам.
- **Алерт на противоречие с ранее устоявшимся фактом.** Если новый claim жёстко противоречит
  claim с высокой историчной уверенностью — это сигнал либо о дезинформации, либо о реальном
  переломе тренда. Достойно отдельного уведомления, а не тихой перезаписи.
- **Дедупликация на уровне смысла, а не текста.** Один и тот же факт может прийти с разными
  формулировками от Agent 1 (текст) и Agent 2 (видео-транскрипт) — нужно схлопывать по смыслу
  (embedding similarity + LLM-judge подтверждение), а не по точному совпадению текста.

Первые два пункта — расширение возможностей, не обязательное для MVP. Третий и четвёртый —
по сути естественная часть §2.2/§2.4 исследования, стоит закладывать сразу.

### Решено: что в MVP, что в v2

- **Дедупликация по смыслу** и **алерт на противоречие с устоявшимся фактом** — в MVP, это
  естественная часть работы "сверка фактов", без них сверка неполная.
- **Рейтинг источников** и **отслеживание нарратива во времени** — сам *алгоритм* (кто из
  источников надёжнее, как менялся нарратив) откладывается на v2, потому что ему нужны недели
  накопленной истории, чтобы от чего-то отталкиваться. Но **накопление данных, на основе которых
  этот алгоритм потом будет считаться, идёт с первого дня MVP** — схема `entities/claims/sources`
  и так сохраняет каждый факт с привязкой к источнику и времени (§2.1 исследования). В v2 не нужно
  ничего досоздавать по хранению — только написать алгоритм поверх уже накопленных данных.

### Новая идея: адаптивная частота анализа (v2)

Раз Агент 3 работает батчем (утренний прогон по всем источникам) и копит историю claims, со
временем он может сам оценить: даёт ли ежедневный анализ значимо новую картину, или тренды
меняются медленнее — и тогда можно перейти на 1-2 прогона в неделю без потери охвата ключевых
изменений, но с экономией. **Обязательное условие: Агент 3 только предлагает и предупреждает, но
не переключает частоту сам** — финальное решение всегда за пользователем (тот же принцип, что и
у порога эскалации $0.10). Реализуемо только в v2, так как требует недель накопленной истории,
чтобы обоснованно предложить переход.

## 2. Технический стек — сводка (детали в 1/2)

Коротко, без повтора обоснований (см. `1. Идея.md` §"Технологический стек" и
`2. Исследование.md` §2.1/2.5):

| Слой | Выбор |
|---|---|
| Оркестрация | LangGraph (`Send` API для параллельного анализа + reducer) |
| LLM | OpenRouter → Claude (haiku для рутины, sonnet для глубокого синтеза) |
| Векторная память | pgvector в Supabase, единая эмбеддинг-модель под все модальности (Gemini Embedding 2) |
| Явные связи/факты | Таблицы `entities/claims/sources` в том же Postgres (не отдельная графовая БД) |
| Очереди/кэш | Redis, общий контейнер `marketing-agency-redis` |
| Контроль качества | LLM-as-judge (структурированный вывод) + FrugalGPT-каскад, порог эскалации $0.10 |
| Точечная перепроверка сырых медиа | video-pipeline MCP (из проекта Hermes) |

## 3. Взаимодействие и уведомления

Агент 3 — внутреннее звено цепочки, не имеет собственного пользовательского интерфейса
(в отличие от Агента 1 с его Telegram-ботом). Но по решению из `1. Идея.md` ему всё равно нужно
иногда достучаться до пользователя — когда стоимость повторного разбора превышает $0.10 или это
уже вторая попытка подряд.

**Открытый вопрос:** через что именно слать это уведомление? Варианты:
- Переиспользовать существующего Telegram-бота Агента 1 (он уже умеет отправлять сообщения
  пользователю) — Агент 3 кладёт уведомление в общую очередь/таблицу, бот Агента 1 его забирает
  и отправляет.
- Агент 3 сам держит прямой вызов Telegram Bot API (свой токен или тот же бот, но без зависимости
  от кода Агента 1).

Первый вариант меньше дублирования (один бот на все агенты), второй — меньше связанности
(Агент 3 не зависит от того, жив ли процесс Агента 1).

**Решено (предварительно):** вариант Б — прямой вызов Telegram Bot API, не завязанный на процесс
Агента 1. Причина: уведомление о превышении порога эскалации — блокирующая ситуация (контент ждёт
решения пользователя), делать её зависимой от того, жив ли соседний процесс, рискованно. Возможно
пересмотрим при дальнейшей разработке.

## 4. Риски и слабые места

| Риск | Вероятность | Последствие | Смягчение |
|---|---|---|---|
| LLM-judge галлюцинирует "противоречие" там, где его нет | Средняя | Ложные алерты, потеря доверия к системе | Self-consistency (несколько сэмплов + голосование, §2.2 исследования) вместо одного вызова судьи |
| Затраты на эскалацию растут быстрее, чем ожидалось (много low-confidence айтемов от Агента 2) | Средняя | Незапланированные расходы | Порог $0.10 + лимит 1 автоповтор на item уже ограничивает верхнюю границу; нужен дневной/недельный cost cap как доп. предохранитель |
| LangGraph как единая точка отказа (граф падает целиком) | Низкая | Простой всего анализа | Чекпоинты состояния графа (LangGraph поддерживает persistence) — не начинать анализ с нуля при сбое |
| Agent 3 — единая точка узкого горла между Agent 1/2 и Agent 4 (оба апстрима сходятся в одном месте) | Средняя | Задержка всей цепочки при перегрузке | Уже частично закрыто паттерном буферизации (пришло от Агента 2) — очередь, а не блокирующий вызов |
| Дрейф калибровки confidence со временем (модели меняются, пороги устаревают) | Низкая на старте, растёт со временем | Эскалация либо избыточна, либо недостаточна | Осознанно отложено в `2. Исследование.md` до v2 (формальная калибровка) — на старте ручная точечная проверка выборки |

## 5. Решено

1. **Уведомления пользователю** — прямой вызов Telegram Bot API (вариант Б), см. §3.
2. **Доп. возможности из §1** — дедупликация и алерт на противоречие в MVP; рейтинг источников и
   отслеживание нарратива — алгоритм в v2, но накопление данных для него идёт с первого дня MVP
   (см. §1 "Решено: что в MVP, что в v2").
3. **Cost cap** — простой предохранитель на MVP: если суммарная трата на автоповторы за один
   утренний прогон достигает **$5**, дальнейшие автоповторы в этом прогоне останавливаются и
   пользователю приходит уведомление. Не жёсткий дневной/недельный лимит — ограничение на прогон.
4. **Адаптивная частота анализа** (реже, чем раз в день, если история покажет, что это не теряет
   охват трендов) — идея зафиксирована в §1 как v2-фича, требует накопленной истории.
