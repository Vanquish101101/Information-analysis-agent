# 1. Идея проекта — Information Analysis Agent (Агент 3)

## Что это

Агент 3 — третье звено в цепочке из 4 агентов маркетингового проекта. Его задача — не искать
и не парсить информацию самому, а **понять смысл и логику** уже собранных данных: найти связи,
выделить главное, оценить достоверность, склеить разрозненные куски в цельную картину и передать
результат дальше в готовом для использования виде.

```
Агент 1 (Intelligence)  →  Агент 2 (Deep parsing)  →  Агент 3 (Information analysis)  →  Агент 4 (Content creation)
   поиск источников          глубокий разбор              синтез смыслов, логика,           готовый контент
   и быстрые факты          контента (видео/аудио/         понимание данных,
                             документы/изображения)         контроль качества
```

Это не моя формулировка — роль Агента 3 в этом виде уже зафиксирована в документах Агента 1
(`05. Архитектура.md` §1.5/§9.1, `04. ТЗ.md` §6.2) и Агента 2 (`1. Идея.md`, `8. Архитектура.md`),
и в схеме Supabase уже зарезервирована пустая схема `information_analysis_agent` в общем проекте
`Marketing agency` (id `wklecdbujgdwnbmfmggi`, eu-central-1).

## Откуда берётся вход

**Agent 2 отдаёт результат напрямую Агенту 3, в обход Агента 1.** Это уже реализованный паттерн
(`handoffToAgent3()` в `Code/src/router/index.js` Агента 2) — сейчас это заглушка, которая всегда
падает, буферизует результат в Supabase (`deep_parsing_agent.agent3_handoff_queue`) и ждёт, пока
Агент 3 не появится. Контракт входных данных от Агента 2 уже описан (`8. Архитектура.md` §7.2):

```json
{
  "job_id": "uuid",
  "content_type": "video | audio | text | document | image",
  "result": { "...": "структура своя для каждого типа контента" },
  "confidence": { "level": "высокая|средняя|низкая", "explanation": "текст" },
  "meta": { "tools_used": [...], "duration_sec": 42, "cost_usd": 0.031, "cache_hit": false }
}
```

Поле `confidence` добавлено намеренно — «чтобы Агент 3 понимал, насколько можно доверять данным»
(прямая цитата из ТЗ Агента 2, §4.4). Более того, в брейншторме Агента 2 (`5. Брейншторм.md`, §0)
заранее оставлена заметка для Агента 3: **Агент 3 должен сам проверять качество распарсенных
данных — сверять, оценивать и решать, передать ли задачу на повторную обработку другому
инструменту, либо сообщить пользователю, что качественно распарсить не удалось.** Это не побочная
функция, а явно заложенная обязанность — контроль качества внутри самого Агента 3, а не только
пассивное потребление confidence-метки.

**С Агентом 1 контракт уже реализован (подтверждено 2026-07-07 чтением исходников).** Таблица
`intelligence_agent.search_results` создана (миграция `002_search_results.sql`), оркестратор
Агента 1 (`Code/src/orchestrator/index.js`) заполняет её после каждого поиска. Точная схема строки:

```sql
id, job_id (uuid), telegram_id, task_type ('report'|'trends'|'search'), query_topics (text[]),
depth, status ('ok'|'partial'|'error'),
result JSONB  -- { raw: { perplexity, youtube, firecrawl } }
telegram_text TEXT,  -- копия Telegram-отчёта, fallback
confidence JSONB,    -- { level: 'высокая|средняя|низкая', explanation }
meta JSONB,          -- { tools_used, cost_usd, duration_sec, model_used }
created_at
```

Агент 3 читает из этой таблицы напрямую через Supabase SDK — по `telegram_id`, сортировка
`created_at DESC` (паттерн чтения, зафиксированный самим Агентом 1 в его задачнике). Поле
`result.raw` — основной источник структурированных данных для анализа; `telegram_text` — fallback
только на случай, если `result.raw` почему-то пуст (не ожидается в норме).

**Статус:** реализация есть, но финальное ручное тестирование со стороны Агента 1 ещё не отмечено
выполненным (см. `Intelligence agent/План разработки/06. Задачник.md` §1.11) — не блокирует
разработку Агента 3, но стоит перепроверить перед продакшеном, что данные реально приходят
корректно.

## Что Агент 3 должен делать

Из первоначального брифа проекта (`История запросов.txt` Агента 2, исходная формулировка задачи)
Агент 3 должен обеспечивать «полное понимание и склейку полученной информации на более глубоком
уровне — с выделенными смыслами», результат — либо краткий дайджест, либо (по запросу) подробный
разбор. Собирая это с явными обязанностями из документов Агента 2, получается три слоя работы:

1. **Контроль качества входа** — проверка confidence-меток и самих данных от Агента 2 (и в
   перспективе Агента 1); при низком доверии — либо запрос на повторный парсинг (например, через
   video-pipeline MCP напрямую), либо явное предупреждение вниз по цепочке / пользователю.
2. **Семантический синтез** — склейка кусков от разных модулей (видео + аудио + текст + документы)
   в единую картину: находить связи, противоречия, повторы, выделять главное, строить логику
   («что из этого следует», «что подтверждено несколькими источниками, а что — нет»).
3. **Подготовка выхода для Агента 4** — результат должен быть пригоден для генерации контента:
   не сырой анализ ради анализа, а структурированные смыслы + at-request подробности.

## Что Агент 3 отдаёт на выход

Контракта Агент3 → Агент4 в документах **не существует вообще** — ни один из трёх агентов его не
описывает. Это открытая точка, которую предстоит спроектировать в этом же проекте (см. вопросы
ниже). По аналогии с уже сложившимся паттерном (job_id/result/confidence/meta) логично, что Агент 3
будет отдавать похожую по духу структуру — сжатый смысловой дайджест + доступ к деталям по запросу —
но точный формат зависит от того, что нужно Агенту 4.

## Технологический стек и ресурсы

### Уже есть и проверено в работе (переиспользовать, не изобретать заново)

- **Supabase** — общий проект `Marketing agency`, схема `information_analysis_agent` уже
  зарезервирована (пустая). Присоединяться к нему, а не поднимать свою БД.
- **Redis** — общий контейнер `marketing-agency-redis`, сеть `marketing-agency-net`.
- **LLM-доступ через OpenRouter** — уже подключён и используется Агентом 1 (`claude-haiku-4-5` для
  быстрых задач, `claude-sonnet-4-6` для глубоких) — тот же маршрут логично использовать и здесь
  для смыслового синтеза.
- **Уже оплаченные и рабочие ключи**: Gemini, Groq, HuggingFace, AssemblyAI, TwelveLabs, Deepgram,
  OpenAI, Apify, Firecrawl, Perplexity — задокументированы и проверены Агентом 2
  (`4. Стек ресурсов и агентов.md` §1.1).
- **video-pipeline MCP-сервер** (Python/FastMCP, из отдельного проекта `Hermes-DeepSeek-v4-Telegram`)
  — 7 готовых инструментов (`download_video`, `get_youtube_transcript`, `extract_audio`,
  `extract_frame`, `detect_scenes`, `transcribe_audio`, `ocr_image`), уже отлажен и работает через
  подпроцессы (баг с зависанием тяжёлых ML-библиотек внутри MCP-сервера найден и исправлен). Полезен
  Агенту 3 для точечной повторной проверки сырого материала, если Агент 2 вернул низкий confidence.
- **Паттерн буферизации при недоступности получателя** — уже реализован в Агенте 2
  (`handoffToAgent3`, сравнение времени ожидания с ожидаемой длительностью, уведомление в Telegram
  при задержке). Тот же паттерн стоит заложить и на стороне Агента 3 → Агент 4.
- **FrugalGPT-style каскад «дешёвый → дорогой инструмент» с эскалацией** — уже спроектирован и
  используется Агентом 2 для парсинга по типам контента; тот же принцип подходит для оценки
  качества/уверенности внутри Агента 3 (например, сначала быстрая дешёвая модель-судья, эскалация
  на более сильную модель только при неуверенности).

### Чего в стеке пока нет — предлагаю варианты

- **Векторная память / семантический поиск.** Для «понимания смысла и логики» и склейки данных из
  разных источников почти наверняка понадобится векторное хранилище (embeddings + similarity
  search) — искать похожие/связанные куски информации, находить противоречия и повторы. В планах
  Агента 1 фигурируют ChromaDB/Qdrant/Pinecone, но ни один из них реально не установлен и не
  оплачен. **Предлагаю pgvector-расширение прямо в уже используемом Supabase** — не нужен новый
  сервис, не нужны новые ключи, данные и так уже там. Отдельный Qdrant/Pinecone имеет смысл только
  если объём/нагрузка вырастут настолько, что Postgres перестанет справляться — маловероятно на
  старте.
- **Оркестрация последовательных и параллельных под-агентов анализа.** В брифе явно требуется
  логика «последовательной и параллельной работы» внутри Агента 3. В соседнем проекте
  `Hermes-DeepSeek-v4-Telegram/Departments` есть два готовых подхода к оркестрации нескольких
  агентов — **оба сейчас деактивированы самим пользователем**: (а) тяжёлый вариант «оркестратор +
  30 специалистов» через `.claude/agents/`, (б) лёгкий вариант «один сильный исполнитель берёт
  оптимальный инструмент по типу и стоимости» (философия, к которой пользователь пришёл после
  отказа от LangGraph/LangSmith). Для Агента 3 это открытый архитектурный вопрос — см. ниже.

## Решение: оркестрация — LangGraph

Для последовательной и параллельной работы под-агентов анализа внутри Агента 3 используется
**LangGraph** — граф-фреймворк даёт явную, читаемую схему потока (узлы = шаги анализа, рёбра =
последовательные/параллельные переходы), что лучше подходит под требование «последовательной и
параллельной логики анализа» из исходного брифа, чем самодельная асинхронная оркестрация.

Ключи уже есть и не нужно ничего регистрировать заново — найдены в
`C:\Users\Unknown\AppData\Local\hermes\.env`: `LANGGRAPH_API_KEY`, `LANGGRAPH_SERVICE_KEY`
(и рядом `LANGSMITH_API_KEY`/`LANGSMITH_SERVICE_KEY` — трассировка вызовов, тот же tenant/org).
В мастер-файле `Project settings\API Keys.txt` этих ключей нет — единственный источник сейчас это
Hermes `.env`, стоит перенести их (или ссылку на них) в мастер-файл, чтобы не зависеть от другого
проекта.

**Важная оговорка из истории Hermes:** LangGraph там ранее деактивировали не из-за проблем с самим
графом, а из-за 403/тарифного ограничения **LangGraph Platform** — их хостинговой платформы для
деплоя графов. Для Агента 3 это не проблема: граф можно запускать как библиотеку прямо внутри
собственного бэкенда (Node/Python-процесс Агента 3), без обращения к LangGraph Platform. Хостинг
через их платформу не требуется.

## Решение: формат выхода для Агента 4

**Оба варианта.** По умолчанию Агент 3 отдаёт краткий структурированный дайджест — набор
атомарных утверждений вида `{ факт, источник, confidence, id_для_деталей }` (не пересказ своими
словами, а сжатая форма тех же конкретных фактов). Если дайджеста недостаточно (Агенту 4 нужна
точная цифра, цитата, полный контекст по конкретному пункту), он запрашивает полный разбор по
`id_для_деталей` — это не пересчёт, а чтение уже посчитанного и сохранённого результата
(Supabase, тот же принцип `cache_hit`, что уже использует Агент 2). Токены/деньги тратятся один
раз на сам анализ; последующий доступ к деталям — это просто чтение из БД.

## Решение: глубина контроля качества

**Гибрид по порогу стоимости**, а не бинарный выбор. `confidence` — это оценка Агента 2, насколько
он сам доверяет результату конкретного разбора (не оценка важности контента), с пояснением причины
(например: аудио было нечётким, OCR распознал текст не полностью, сцены видео не удалось чётко
определить). Повторный разбор при низкой confidence обычно означает вызов следующей, более дорогой
ступени каскада Агента 2 (более тяжёлая модель транскрибации, платный сервис вроде
TwelveLabs/AssemblyAI и т.п.) — то есть реальные деньги, но по-разному в зависимости от типа и
объёма контента.

Правило:
- Ожидаемая стоимость повторного разбора **≤ $0.10** за item → Агент 3 сам решает и запускает
  повтор автоматически, без вопросов.
- Стоимость **> $0.10**, или это уже вторая попытка подряд по этому же item (лимит — 1 автоповтор,
  чтобы не зациклиться) → Агент 3 останавливается и спрашивает пользователя через Telegram
  (тот же канал уведомлений, что уже используется Агентом 2 для задержек), объясняя: что не удалось
  разобрать, почему, и сколько будет стоить повтор.
