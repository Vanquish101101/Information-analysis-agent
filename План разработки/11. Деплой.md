## 2. Деплой Агента 3 (MVP)

### 2.1 Порядок первого деплоя

```
1. Подключиться к серверу по SSH
   ssh root@<hetzner-ip>

2. Склонировать репозиторий
   git clone https://github.com/<username>/information-analysis-agent.git
   cd information-analysis-agent

3. Создать .env файл
   cp .env.example .env
   nano .env      ← заполнить ключи (см. шаблон в "8. Разработка.md" §2.2)

4. Применить миграцию схемы Supabase
   (через Supabase SQL editor или CLI — таблицы entities/claims/sources/runs/pending_user_decisions,
   см. "7. Архитектура.md" §8.2)

5. Запустить контейнер
   docker compose up -d

6. Проверить статус
   docker compose ps
   docker compose logs -f

7. Раскомментировать подключение Агента 3 в главном деплой-файле
   (Marketing agency Project/Инфраструктура (Docker)/docker-compose.yml, строка include:)
```

### 2.2 Проверка после деплоя

- [ ] Контейнер в статусе `running`
- [ ] MCP-сервер отвечает на `analysis_status()` (проверить вызовом от Агента 4 или тестовым MCP-клиентом)
- [ ] Планировщик стартовал и логирует наблюдение за очередями с 08:00
- [ ] Таблицы схемы `information_analysis_agent` созданы и доступны
- [ ] Redis доступен: `docker exec -it redis redis-cli ping` → PONG
- [ ] Тестовое Telegram-уведомление реально доставляется (проверить принудительным триггером эскалации)
- [ ] Логи не содержат критических ошибок

## 3. Обновление агента

Тот же паттерн, что у Агента 1 и Агента 2 (единообразие по всей цепочке):

```bash
cd information-analysis-agent
git pull origin main
docker compose up -d --build
```

## 4. Мониторинг и логи

| Инструмент | Что мониторит | Как смотреть |
|---|---|---|
| **LangSmith** | Трейсинг узлов LangGraph, вызовы LLM | langsmith.com → dashboard |
| **Helicone** | Расходы по каждому LLM-вызову, сверка с cost cap $5/прогон | helicone.ai → dashboard |
| **Portainer** | Статус контейнера, CPU/RAM | Общий, как у других агентов |
| **Docker logs** | Планировщик, граф, MCP-сервер, эскалации | `docker compose logs -f` |
| **Supabase `runs`** | История прогонов, стоимость, число обработанных claims | SQL-запрос / дашборд Supabase |

## 5. Резервное копирование

| Данные | Где хранится | Периодичность |
|---|---|---|
| `entities/claims/sources/runs` | Supabase (PostgreSQL) | Ежедневно (автоматически Supabase) |
| `.env` | Сервер | Отдельная зашифрованная копия — тот же принцип, что у Агента 1/2 |
| Логи | Docker logs | 30 дней (ротация) |

Специфичного для Агента 3 бэкапа сверх общего принципа не требуется — данные и так в общем Supabase
с его штатным резервным копированием.
