## 1. Принципы архитектуры

| Принцип | Что это означает на практике |
|---|---|
| **Событийность, не расписание** | Батч-анализ стартует по факту готовности данных (очередь "успокоилась"), а не по жёсткому времени — см. разд. 6.1 |
| **Дёшево — сам, дорого/важно — спроси** | Автоповтор при confidence ≤ порога и cost ≤ $0.10 — автоматически; дороже или спорнее — вопрос пользователю в Telegram |
| **Pull, не push, на выходе** | В отличие от Агента 2 (который сам "толкает" результат Агенту 3), Агент 3 отдаёт результат только по запросу через MCP — недоступность Агента 4 не блокирует и не требует буфера с retry на этой стороне |
| **Единый язык стека** | Node.js (ESM) + `@langchain/langgraph`, консистентно с Агентом 1 и 2 |
| **Разделение ответственности** | Агент 3 не ищет (Агент 1) и не парсит сырой контент (Агент 2) — только сверка, синтез и контроль качества уже готовых данных |
| **Наблюдаемость** | Все LLM-вызовы через Helicone; трейсинг графа через LangSmith; стоимость каждого прогона — в `runs` |

## 3. Контейнеризация (Docker)

### 3.1 Принцип

Агент 3 — отдельный контейнер (образ `information-analysis-agent`), подключённый к уже
существующей общей инфраструктуре, как и остальные агенты цепочки.

### 3.2 Docker Compose Агента 3 (черновик, уточняется на этапе Разработки)

```yaml
# Information analysis agent/Code/docker-compose.yml (появится на этапе "8. Разработка")
name: information-analysis-agent

services:
  information-analysis-agent:
    build: .
    image: information-analysis-agent:v0.1.0
    container_name: information-analysis-agent
    restart: unless-stopped
    env_file: .env
    volumes:
      - ./logs:/app/logs
    networks:
      - marketing-agency-net   # общая сеть — доступ к Redis/Supabase по общей инфраструктуре

networks:
  marketing-agency-net:
    external: true            # сеть уже создана в "Инфраструктура (Docker)/docker-compose.yml"
```

Никаких тяжёлых Python-зависимостей внутри образа не нужно — вся тяжёлая работа (эмбеддинги, LLM,
точечная перепроверка медиа) идёт через внешние API/MCP (video-pipeline — отдельный уже
существующий сервис в проекте Hermes, не часть этого образа).

### 3.3 Подключение к общей инфраструктуре

- Добавляется в главный деплой-файл (`Marketing agency Project/Инфраструктура (Docker)/
  docker-compose.yml`) через `include:` — раскомментировать строку, когда контейнер будет готов.
- До этого — разрабатывается и тестируется через собственный `docker compose up` с
  `networks.marketing-agency-net.external: true`, не трогая мастер-файл.

## 4. MCP-протокол (входящий и исходящий)

### 4.1 Агент 3 как MCP-сервер (входящие вызовы от Агента 4 / любого MCP-клиента)

```
analysis_digest(run_id?)
  → { digest_id, run_at, facts[], contradictions[], meta }   — по умолчанию последний прогон

analysis_detail(claim_id)
  → { claim_id, statement, sources[], reasoning, history }

analysis_status()
  → { last_run_at, status, items_processed, cost_usd, pending_user_decisions[] }
```

Точные схемы — см. `5. ТЗ.md` §3.2. **Обновлено (Слайс 11, коммит `71efe4a`, v0.7.0):** модель
гибридная push+pull, не чистый pull. Содержимое дайджеста Агент 4 всё равно забирает сам через
`analysis_digest(run_id)` — MCP-вызов не заменён. Но после каждого прогона Агент 3 дополнительно
толкает лёгкое уведомление-указатель (`run_id`, без самого дайджеста) через Redis pub/sub
(`notifications:agent4`, best-effort) и надёжную таблицу `information_analysis_agent.
agent4_handoff_queue` (с ретраями) — чтобы Агент 4 не был обязан поллить `analysis_status()` вслепую
и знал, когда именно появился новый дайджест. См. `Code/src/handoff/agent4Handoff.js`.

### 4.2 Агент 3 как MCP-клиент (исходящие вызовы)

- **video-pipeline MCP** (проект Hermes) — точечная эскалация при низком confidence и
  cost ≤ $0.10 (`download_video`, `transcribe_audio`, `ocr_image` и т.д. по необходимости).
- Никаких других исходящих MCP-вызовов на MVP не требуется — Агент 3 не является клиентом Агента 1
  или Агента 2 (он их данные читает напрямую из общих очередей/таблиц, не через MCP-вызов к ним).

### 4.3 Схема вызовов в цепочке

```
[Агент 2] ──пишет напрямую──▶ [agent3_handoff_queue, Redis/Supabase]
[Агент 1] ──пишет напрямую──▶ [intelligence_agent.search_results] (когда реализовано)
                                          │
                                          ▼
                              [Агент 3 — Планировщик читает очереди]
                                          │
                                          ▼
                              [Граф анализа → результат в Supabase]
                                          │
                                          ▼
                    [globalSynthesis: digest сохранён] ──push (run_id)──▶ [notifications:agent4
                                          │                                 + agent4_handoff_queue]
                                          ▼
[Агент 4] ──MCP: analysis_digest(run_id)──▶ [Агент 3 отвечает из уже посчитанного]
```

## 5. Граф анализа (LangGraph)

### 5.1 Узлы графа

```
Dispatcher
    │  собрать job'ы прогона, разбить на независимые куски
    ▼
Send × N (параллельно, по job'у/кластеру)
    │  извлечь claims {subject, predicate, object, qualifiers} из result
    ▼
Reducer
    │  собрать все claims прогона воедино
    ▼
Дедупликация (embedding similarity ≥ 0.85 + LLM-judge подтверждение)
    ▼
Детекция противоречий (LLM-as-judge, self-consistency 3 сэмпла для спорных случаев)
    ▼
Контроль качества / Эскалация (conditional edge по confidence + cost, см. разд. 6.2)
    ▼
GlobalSynthesis (claude-sonnet-4-6: дайджест + количественные агрегаты)
    ▼
Запись результата (Supabase: entities/claims/runs)
```

### 5.2 Персистентность

LangGraph checkpoints сохраняют состояние после каждого узла — при сбое процесса прогон
возобновляется с последнего пройденного узла, не с нуля (смягчение риска "единая точка отказа" из
`3. Брейншторминг.md` §4).

## 6. Отказоустойчивость

### 6.1 Планировщик батча (событийный триггер)

```
С 08:00 — периодическая проверка очередей (agent3_handoff_queue + search_results)
        ↓
Новых элементов нет 15 минут подряд?
  Да  → батч считается готовым → запуск графа анализа
  Нет → продолжить наблюдение
        ↓
Достигнут потолок 11:00 (3 часа с 08:00)?
  Да  → принудительный запуск с тем, что накоплено → пометить недостающее в meta →
        уведомить пользователя, что часть данных не вошла в прогон
```

### 6.2 Контроль качества / эскалация (внутри графа)

```
Claim с confidence "низкая" от Агента 2/1
        ↓
Ожидаемая стоимость повторного разбора через video-pipeline MCP?
  ≤ $0.10 И не было автоповтора для этого job'а → авто-эскалация (1 попытка)
  > $0.10 ИЛИ уже был автоповтор → в pending_user_decisions + Telegram-уведомление
        ↓
Суммарная трата на автоповторы за прогон ≥ $5?
  Да → остановить дальнейшие автоповторы в этом прогоне, уведомить, продолжить синтез с тем, что есть
```

### 6.3 Недоступность Агента 4

**Обновлено (Слайс 11):** сам дайджест по-прежнему не требует буфера/retry — модель забора
данных осталась pull (`analysis_digest()`/`analysis_status()`), результат последнего успешного
прогона всегда доступен, даже если новый прогон не завершён или упал. Но лёгкое уведомление-
указатель о готовности прогона (см. §4.1/§4.3) — это push, и для него буфер/retry уже есть:
надёжный слой — `agent4_handoff_queue` с ретраями `[500, 2000, 8000]`мс на INSERT
(`Code/src/handoff/agent4Handoff.js`), быстрый слой — Redis pub/sub best-effort (сообщение может
потеряться при недоступности Redis без последствий, т.к. это не единственный источник истины —
Агент 4 может при старте/по таймеру просто прочитать `agent4_handoff_queue` напрямую).

## 8. Хранилище данных

### 8.1 Redis (общий, `marketing-agency-net`) — быстрый слой

| Назначение | Ключ | TTL |
|---|---|---|
| Состояние планировщика (последний замеченный элемент очереди, таймер тишины) | `scheduler:agent3:state` | До следующего прогона |
| Чекпоинты LangGraph (промежуточное состояние текущего прогона) | `checkpoint:agent3:{run_id}` | До завершения прогона |

### 8.2 Supabase (общий проект `Marketing agency`, схема `information_analysis_agent`) — надёжный слой

```
information_analysis_agent.entities
  id, name, type, first_seen_at, last_seen_at, embedding (pgvector)
  -- embedding нужен для сопоставления claim.subject/object с уже известной сущностью
  -- при извлечении (entity resolution), отдельно от дедупликации claims целиком

information_analysis_agent.sources
  id, agent (1|2), source_type, ref, raw_job_id, created_at

information_analysis_agent.claims
  id, subject_entity_id, predicate, object_entity_id, object_value,
  confidence_level, confidence_explanation, source_id, extracted_at, superseded_by,
  embedding (pgvector)

information_analysis_agent.runs
  id, run_at, status, cost_usd, items_processed, escalations_auto, escalations_pending_user

information_analysis_agent.pending_user_decisions
  id, job_id, question, estimated_cost_usd, status, created_at, resolved_at
```

Таблицы создаются на этапе «8. Разработка» — схема `information_analysis_agent` уже зарезервирована
(см. `1. Идея.md`).

## 9. Безопасность

- API-ключи (OpenRouter, Gemini, LangGraph/LangSmith, Telegram) — в `.env`, не в коде/репозитории,
  тот же принцип, что у Агента 1 и 2.
- Агент 3 не открыт наружу — доступен только внутри `marketing-agency-net`, вызывается через MCP
  Агентом 4 (или другим внутренним клиентом).
- Supabase — доступ по service-ключу для операций записи, без публичного анонимного доступа к
  схеме `information_analysis_agent`.
- Telegram-токен для прямых уведомлений — источник (мастер-файл `API Keys.txt` или отдельная
  регистрация) уточняется на этапе Разработки (открытый момент из `4. Технологический стек.md`).
