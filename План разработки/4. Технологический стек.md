# 4. Технологический стек — Information Analysis Agent (Агент 3)

Свод всех решений по стеку из `1. Идея.md`, `2. Исследование.md`, `3. Брейншторминг.md` в один
документ + несколько решений, которые раньше не были явно зафиксированы (язык рантайма, MCP-интерфейс
Агента 3, наблюдаемость).

---

## 1. Рантайм и язык

**Решение: Node.js (ESM), тот же стек, что у Агента 1 и Агента 2.** Оба соседних агента уже на
Node.js (`@modelcontextprotocol/sdk`, `@supabase/supabase-js`, `ioredis`, `openai`) — если Агент 3
тоже на Node.js, весь пайплайн говорит на одном языке, проще переиспользовать паттерны (буферизация,
tool-registry, MCP-обёртки) и не нужен межъязыковой мост между агентами.

LangGraph при этом не заставляет переходить на Python — есть официальный JS-пакет
`@langchain/langgraph` с тем же графовым движком и `Send` API для параллельного map-reduce
(§2.4 исследования). Используем его, а не Python-версию.

**Если понадобится тяжёлая ML-библиотека** (например, свой embedding-пересчёт вне API) — паттерн
уже есть у Агента 2: Python-сайдкар отдельным MCP-сервером (как `video-pipeline`), а не переписывание
всего агента на Python. Пока для Агента 3 такой необходимости не видно — вся тяжёлая работа
(эмбеддинги, LLM) идёт через внешние API.

## 2. Оркестрация

- **LangGraph.js** — граф узлов для последовательного и параллельного анализа. Ключевой паттерн:
  `Dispatcher → N параллельных Send → PerItemAnalysis → Reducer → GlobalSynthesis` (§2.4
  исследования).
- **Персистентность графа (checkpoints)** — LangGraph поддерживает сохранение состояния графа;
  использовать для устойчивости к сбоям (риск "LangGraph как единая точка отказа" из брейншторма,
  §4) — не начинать анализ заново при падении процесса.
- **LangSmith** — трейсинг вызовов графа. Ключи уже есть в `C:\Users\Unknown\AppData\Local\hermes\
  .env` (`LANGSMITH_API_KEY`, `LANGSMITH_SERVICE_KEY`), тот же tenant/org, что и `LANGGRAPH_API_KEY`
  /`LANGGRAPH_SERVICE_KEY`. Не обязателен для MVP, но раз ключи уже есть — подключить не будет
  стоить дополнительной регистрации.
- **LangGraph Platform (хостинг) не используется** — граф исполняется как библиотека внутри
  собственного Node.js-процесса Агента 3 (уточнение из `1. Идея.md`: LangGraph Platform ранее
  деактивирована в Hermes из-за 403/тарифа, но это не блокирует использование графа как библиотеки).

## 3. LLM-доступ

- **OpenRouter** как транспортный слой — тот же маршрут, что уже использует Агент 1.
  - `claude-haiku-4-5` — рутинные операции (LLM-as-judge на структурированных полях, дешёвый тир
    каскада).
  - `claude-sonnet-4-6` — глубокий семантический синтез (GlobalSynthesis-узел, финальная сборка
    дайджеста).
- **Helicone** — прокси для наблюдаемости $/токены. Ключи уже есть и используются Агентом 1 как
  "источник истины" по расходам (`config.yaml` + `DEEPSEEK_BASE_URL` паттерн). Рекомендую завести
  Агента 3 через тот же прокси с первого дня — иначе cost cap $5/прогон (решение из брейншторма,
  §5) нечем будет измерять точно.

## 4. Хранилище

### Supabase (Postgres) — общий проект `Marketing agency`, схема `information_analysis_agent`

- **pgvector** — векторная память. Эмбеддинг-модель — **Gemini Embedding 2** (не отдельная модель
  под каждый тип контента) — единое векторное пространство для текста/видео/аудио-описаний,
  найдено в исследовании Агента 2 (Haystack + Gemini Embedding 2, `2. Исследование.md`, Часть 1
  "От Агента 2"). Ключ Gemini уже оплачен и используется другими агентами цепочки.
- **`entities` / `claims` / `sources`** — обычные реляционные таблицы (не Apache AGE — ненадёжен на
  managed Supabase в 2026, `2. Исследование.md` §2.1). `claims` хранит `(subject_entity_id,
  predicate, object_entity_id/value)` + JSONB для провенанса и confidence, с индексами по
  subject/predicate/object. Эта же схема — источник данных для будущего (v2) рейтинга источников и
  отслеживания нарратива (§1 брейншторма) — копится с первого дня MVP, алгоритм поверх неё пишется
  позже.

### Redis — общий контейнер `marketing-agency-redis`, сеть `marketing-agency-net`

- Кэш промежуточных результатов, очередь входящих job'ов от Агента 2 (`agent3_handoff_queue`,
  уже существует на стороне Агента 2) и от Агента 1 (`intelligence_agent.search_results`, когда
  будет реализовано).

## 5. Интеграция с остальной цепочкой

### Вход

- **От Агента 2** — напрямую, контракт уже описан (`8. Архитектура.md` Агента 2, §7.2):
  `{ job_id, content_type, result, confidence: { level, explanation }, meta }`.
- **От Агента 1** — временно разбор готового Telegram-текста, переход на
  `intelligence_agent.search_results` после доработки Агента 1 (см. `Доработки для Агента 1
  (передать).md`).

### Выход — MCP-сервер Агента 3

По аналогии с Агентом 2 (который уже экспонирует `deepparsing_parse`/`deepparsing_status` через
`@modelcontextprotocol/sdk`), Агент 3 тоже должен быть MCP-сервером, а не просто внутренним
скриптом — чтобы Агент 4 и любой другой MCP-совместимый клиент мог его вызывать стандартным
образом:

- `analysis_digest` — вернуть краткий дайджест по задаче/теме (атомарные факты + confidence +
  количественные агрегаты, §2.3 исследования).
- `analysis_detail` — вернуть полный разбор по `id_для_деталей` конкретного пункта дайджеста
  (чтение уже посчитанного, решение из `1. Идея.md`).
- `analysis_status` — статус текущего/последнего прогона (для мониторинга, аналогично
  `deepparsing_status`).

### Точечная перепроверка сырых медиа

- **video-pipeline MCP** (проект `Hermes-DeepSeek-v4-Telegram`) — 7 готовых инструментов, уже
  отлажен. Используется, когда confidence от Агента 2 низкий и порог эскалации ($0.10/item)
  позволяет автоповтор.
