## Часть 2. Внешнее исследование — слой синтеза смысла

### 2.1 GraphRAG и графовое представление знаний

**Microsoft GraphRAG** строит граф сущностей/отношений LLM-экстракцией, кластеризует его в
"сообщества" и суммирует иерархически — даёт качественные ответы на *глобальные* вопросы
("какие тренды доминируют"), но требует дорогой LLM-индексации всего корпуса заранее.
**LazyGraphRAG** (MS Research, 2024) — тот же граф, но без предварительной суммаризации: LLM тратится
только на relevance-тесты по бюджету в момент запроса — сравнимое качество, но **~в 700 раз
дешевле**. Это прямая аналогия уже принятому у нас паттерну "эскалация по бюджету $0.10".

**Про графовую СУБД в Postgres:** Apache AGE (графовое расширение) технически существует, но на
2026 год **официально не поддерживается управляемым Supabase** (обсуждение
supabase/discussions #40285) — пришлось бы кастомно собирать расширение, риск для managed-инстанса.

**Практичная альтернатива** (используется, например, в TrustGraph): реифицированная схема
`entities / claims / sources` обычными таблицами Postgres — `claims` хранит
`(subject_entity_id, predicate, object_entity_id/value)` плюс JSONB для провенанса и confidence, с индексами по
subject/predicate/object. Даёт быстрый доступ "сущность → все связанные факты" без сложных
self-join'ов и без отдельной графовой БД.

### 2.2 Сверка фактов между источниками и детекция противоречий

Классический подход — NLI (natural language inference): классификация пар "факт-факт" как
entailment/contradiction/neutral. На датасетах FEVER/SciFact-Open чистая NLI-классификация даёт
умеренную точность (теряет ≥15 F1 при масштабировании корпуса) — **обучать/дообучать отдельную
NLI-модель невыгодно** при нашем масштабе.

Практичнее — **LLM-as-judge** поверх кластеров фактов (одинаковый subject+predicate, разные
object): модель выдаёт структурированный вердикт `{label, confidence, explanation}`. Для
устойчивости судейства — **self-consistency** (Wang et al., 2022: несколько сэмплов
reasoning-цепочки + голосование большинством) и **LLM fan-out** (несколько параллельных прогонов/
моделей + агрегатор) — оба паттерна естественно ложатся на параллельный fan-out в LangGraph.

Итоговый confidence по факту практично считать как комбинацию: доля источников за/против
(взвешенная по надёжности источника) + средний confidence LLM-судей + энтропия голосов
self-consistency — не одно число, а составная метрика.

### 2.3 Коммерческие продукты (Brandwatch, Talkwalker, Primer.ai, Signal AI)

Внутренние алгоритмы проприетарны, но общий архитектурный паттерн виден: ingestion → нормализация
сущностей/тем → **количественные агрегаты** (объём упоминаний, охват, динамика сентимента во
времени) как числовой каркас, а не только текстовый синтез. Signal AI/Primer.ai используют
event-detection и entity-centric представления.

**Вывод для дизайна digest:** чисто текстовая сверка фактов (как в `1. Идея.md` сейчас) — это
неполное решение по сравнению с индустрией. В дайджест стоит закладывать не только
`{факт, источник, confidence}`, но и количественные метрики: сколько источников подтверждают факт,
суммарный охват (просмотры/лайки из meta Агента 1/2), динамика во времени (появилось только что vs
устойчиво держится N дней).

### 2.4 LangGraph: паттерн параллельного анализа (map-reduce)

LangGraph официально поддерживает нужный паттерн через объект **`Send`**
(docs.langchain.com/oss/python/langgraph/graph-api): узел-диспетчер возвращает список
`Send(node_name, partial_state)`, что запускает целевой узел параллельно для каждого элемента в
рамках одного "супершага" (модель исполнения Pregel — message passing). Структура для Агента 3:

```
Dispatcher → N параллельных Send → PerItemAnalysis (по каждому куску контента) → Reducer → GlobalSynthesis
```

Бюджетную маршрутизацию (эскалация/фолбэк при низком confidence — уже решённый паттерн из
`1. Идея.md`) реализуют conditional edges, читающие поля state вроде `remaining_budget` /
`uncertainty_score`. Это не абстрактная теория — конкретный, задокументированный API LangGraph,
прямо подходящий под уже принятое архитектурное решение.

### 2.5 LLM-as-judge и оценка уверенности у "чёрного ящика"

FrugalGPT (Chen et al., 2023, уже используется Агентом 2) — каскад "дешёвая модель → эскалация"
даёт до 98% экономии без потери качества. Более новая идея unified cascade routing обобщает это в
динамический контроллер моделей вместо жёсткой цепочки.

Ключевая практическая проблема: у Claude через OpenRouter **нет доступа к logprobs**, то есть
классические методы извлечения уверенности "из вероятностей модели" не работают. Подход в духе
**CONSTRUCT (2026)** решает это иначе — оценивает доверие на уровне **отдельных полей
структурированного JSON-вывода**, без дообучения модели. Это прямо ложится на нашу схему `claim`
(`subject/predicate/object/qualifiers`) — можно получить trust-score на каждое поле, а не только
общий скор на весь ответ.

## Рекомендации: что берём, что нет

### Берём

- **Реифицированная схема `entities/claims/sources`** обычными таблицами Postgres (не Apache AGE —
  ненадёжен на managed Supabase) — даёт большую часть пользы графового подхода почти без
  инфраструктурного риска. Дополняет уже решённый pgvector: pgvector — для similarity search,
  claims-таблицы — для явных связей и сверки фактов.
- **LangGraph `Send` API + reducer-узел** для параллельного анализа кластеров фактов — официально
  задокументированный map-reduce паттерн, прямое попадание в уже принятое решение "LangGraph для
  оркестрации" из `1. Идея.md`.
- **LLM-as-judge со структурированным выводом** (`label/confidence/explanation`, в духе CONSTRUCT)
  как основной механизм детекции противоречий между источниками — дешевле и практичнее обучения
  своей NLI-модели.
- **Количественные агрегаты в дайджесте** (число источников, охват, динамика во времени) —
  уточнение к формату выхода для Агента 4 из `1. Идея.md`: дайджест = факты + confidence + эти
  метрики, а не только текстовая сводка.
- FrugalGPT-каскад — уже принят, здесь только дополнительное подтверждение из независимого
  источника.

### Не берём сейчас (overkill)

- **Полный GraphRAG** (community detection + иерархическая суммаризация всего корпуса) — оправдан
  только при по-настоящему больших объёмах и частых "глобальных" запросах по всей истории. На
  старте не нужен; идея LazyGraphRAG (бюджет на relevance-тесты) полезнее как *паттерн эскалации*,
  чем как инструмент целиком.
- **Отдельная графовая СУБД** (Neo4j и т.п.) — лишний компонент при уже принятом Postgres/Supabase.
- **Дообучение своей NLI-модели** на FEVER/SciFact — дорого, даёт умеренный прирост против
  LLM-as-judge на Claude при нашем масштабе.
- **Формальная статистическая калибровка confidence** (по held-out разметке) — правильная цель на
  будущее (v2), но на старте достаточно эмпирических порогов эскалации с точечной ручной проверкой.
