## 1. Принципы тестирования

| Принцип | Что это означает |
|---|---|
| **Тест реальным батчем** | Главная проверка — прогон на реальных данных, накопленных от Агента 1/2 за день, не только синтетика |
| **Тест эскалации осознанно** | Специально подсовывать низкий confidence и проверять оба пути решения (авто vs. вопрос пользователю) |
| **Тест стоимости** | Каждый прогон замеряет `cost_usd`, проверяется срабатывание порога $0.10 и cost cap $5 |
| **Тест устойчивости** | Обрывать процесс посреди прогона, проверять восстановление с checkpoint, а не с нуля |
| **Тест на граничных случаях** | Пустая очередь, один источник, полностью противоречивые данные, `result.raw` пуст |

## 2. Уровни тестирования

### 2.1 Уровень 1 — Unit: отдельные узлы графа и модули

**Dispatcher / Send / Reducer:**
- [ ] Корректно разбивает накопленные job'ы на независимые куски работы
- [ ] Параллельные `Send`-вызовы не блокируют друг друга (реальный параллелизм, не последовательный)
- [ ] Reducer собирает claims со всех параллельных веток без потерь

`[ + ]` — *новые тест-кейсы графа*

**Извлечение claims (PerItemAnalysis):**
- [ ] Из `content_type: video/audio/text/document/image` (вход от Агента 2) корректно извлекает `{subject, predicate, object, qualifiers}`
- [ ] Из `result.raw` Агента 1 (perplexity/youtube/firecrawl) корректно извлекает claims
- [ ] Fallback на `telegram_text`, если `result.raw` пуст — извлекает те же claims из текста

**Дедупликация:**
- [ ] Два одинаковых по смыслу, но по-разному сформулированных claim — similarity ≥ 0.85, LLM-judge
  подтверждает слияние
- [ ] Два похожих, но по сути разных claim (similarity близко к порогу) — LLM-judge отклоняет
  автоматическое слияние
- [ ] Явно разные claims — не попадают в кандидаты на дедуп вообще

**Детекция противоречий:**
- [ ] Два claim с одинаковым `(subject, predicate)`, разным `object` — LLM-judge помечает
  `contradict`
- [ ] Спорный случай (неочевидное противоречие) — включается self-consistency (3 сэмпла), решение
  по большинству
- [ ] Противоречие с claim, у которого исторический confidence "высокий" — попадает в
  `contradictions` дайджеста с explicit explanation, не тихая перезапись

**Эскалация / контроль качества:**
- [ ] confidence "низкая" + ожидаемая стоимость ≤ $0.10 → автоповтор без вопроса
- [ ] confidence "низкая" + стоимость > $0.10 → запись в `pending_user_decisions` + Telegram
- [ ] Повторная попытка для того же job'а (уже был 1 автоповтор) → не повторяет автоматически,
  уходит к пользователю
- [ ] Суммарная стоимость автоповторов за прогон достигает $5 → дальнейшие автоповторы
  останавливаются, уведомление отправлено, синтез продолжается с неполными данными

**GlobalSynthesis:**
- [ ] Дайджест содержит `facts[]` с `confidence`, `sources_count`, `reach_estimate`
- [ ] `contradictions[]` заполнен, если были найдены противоречия
- [ ] `meta` содержит корректные `items_processed`, `cost_usd`, `escalations_auto`/`_pending_user`

### 2.3 Уровень 3 — End-to-End: полный батч-прогон

**Сценарий 1 — Чистый прогон:**
- [ ] Данные без дублей и противоречий → корректный дайджест, `contradictions: []`

**Сценарий 2 — Дубли между Агентом 1 и Агентом 2:**
- [ ] Один и тот же факт пришёл текстом (Агент 1) и из видео-транскрипта (Агент 2) → дедуп
  схлопывает в один claim с двумя источниками

**Сценарий 3 — Противоречие:**
- [ ] Новый claim противоречит устоявшемуся (высокий исторический confidence) → явный алерт в
  дайджесте, не тихая замена

**Сценарий 4 — Дешёвая эскалация:**
- [ ] Низкий confidence, стоимость повтора ≤ $0.10 → автоповтор без участия пользователя

**Сценарий 5 — Дорогая эскалация:**
- [ ] Низкий confidence, стоимость > $0.10 → `pending_user_decisions` + Telegram-вопрос,
  прогон продолжается без блокировки на этом claim

**Сценарий 6 — Cost cap:**
- [ ] Суммарная трата на автоповторы достигает $5 за прогон → дальнейшие автоповторы
  остановлены, синтез идёт с тем, что есть, пользователь уведомлён

**Сценарий 7 — Планировщик: естественная тишина:**
- [ ] Очередь перестаёт пополняться на 15 минут → прогон стартует автоматически

**Сценарий 8 — Планировщик: защитный потолок:**
- [ ] Очередь не "успокаивается" до 11:00 → прогон стартует принудительно, недостающее помечено
  в `meta`, пользователь уведомлён

**Сценарий 9 — MCP-выход для Агента 4:**
- [ ] `analysis_digest()` без параметров → последний прогон
- [ ] `analysis_detail(claim_id)` → полный разбор конкретного факта с цитатами источников
- [ ] `analysis_status()` → корректный статус, включая `pending_user_decisions`

**Сценарий 10 — Пустой `result.raw` от Агента 1:**
- [ ] Fallback на разбор `telegram_text` — граф всё равно извлекает claims, просто менее точно

## 3. Качество дайджеста — критерии оценки

| Критерий | Что проверяем | Проходит если |
|---|---|---|
| **Точность** | Факты в дайджесте соответствуют исходным данным (не искажены синтезом) | Ручная выборочная проверка 100% совпадения смысла |
| **Дедуп-полнота** | Нет очевидных дублей одного и того же факта | 0 необнаруженных дублей в тестовой выборке |
| **Противоречия не потеряны** | Реальные противоречия попадают в `contradictions`, а не молча теряются | 100% обнаружение на тестовом наборе с заведомыми противоречиями |
| **Confidence корректен** | Уровень confidence в дайджесте отражает реальное качество источников | Ручная сверка на выборке |
| **Количественные агрегаты** | `sources_count`/`reach_estimate` присутствуют и осмысленны | Присутствуют у каждого факта |
| **Стоимость прогона** | Общая стоимость батча | В пределах ожидаемого бюджета (уточнить целевую цифру при первых реальных прогонах) |
| **Устойчивость** | Прогон не падает целиком из-за одного проблемного job'а | 1 сбойный job не блокирует остальные |

## 5. Тестирование отказоустойчивости

| Что ломаем | Как проверить | Ожидаемый результат |
|---|---|---|
| Redis недоступен | Остановить контейнер | Планировщик не падает, повторяет попытки, уведомляет при длительной недоступности |
| Supabase недоступен | Неверный ключ в тесте | Прогон не стартует с явной ошибкой, а не тихим зависанием; уведомление пользователю |
| Процесс Агента 3 падает посреди прогона | `kill -9` во время графа | При перезапуске — восстановление с последнего checkpoint, не с нуля |
| OpenRouter/Claude недоступен | Симулировать timeout | Явная ошибка узла, попадает в `meta.errors` дайджеста, не блокирует остальные claims |
| video-pipeline MCP недоступен | Остановить сервер | Эскалация помечается как неудавшаяся, claim остаётся с исходным низким confidence, не блокирует прогон |
| Агент 4 недоступен | Не запускать его при тесте | Никак не влияет на Агента 3 (pull-модель — разд. 6.3 архитектуры) |
