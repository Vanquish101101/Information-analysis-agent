# 5. ТЗ — Information Analysis Agent (Агент 3)

Формальное техническое задание, собранное из решений в файлах `1`–`4`. Где параметр не был явно
зафиксирован раньше (например, конкретный порог similarity для дедупликации) — здесь предложено
разумное значение по умолчанию с пометкой, что его можно пересмотреть на этапе архитектуры/разработки.

---

## 1. Назначение и область действия

Агент 3 — третье звено цепочки из 4 агентов. Принимает структурированные результаты от Агента 1
(поиск) и Агента 2 (глубокий парсинг), производит семантический синтез (сверка фактов, дедупликация,
контроль качества, детекция противоречий) и отдаёт результат Агенту 4 (создание контента) через
собственный MCP-интерфейс.

Работает **батчем** — не по каждому отдельному job'у в реальном времени, а раз в сутки собирает всё
накопленное за день и производит один прогон анализа (экономия по сравнению с точечной обработкой
каждого элемента отдельно, см. `3. Брейншторминг.md` §"Cost cap"). Точечные повторные обращения
(эскалация при низком confidence) — исключение из общего батч-режима, случаются внутри прогона.

## 2. Функциональные требования

### 2.1 Приём и накопление данных

- Слушать/опрашивать очередь входа от Агента 2 (`deep_parsing_agent.agent3_handoff_queue`) и
  (когда будет реализовано) таблицу `intelligence_agent.search_results` от Агента 1.
- Каждый входящий элемент сохраняется как есть (сырой job) до начала батч-обработки — ничего не
  анализируется в момент поступления, только накопление.

### 2.2 Батч-анализ (утренний прогон)

**Триггер — событийный, не по фиксированному времени.** Жёсткое время (например, "09:00") ненадёжно:
если в конкретный день у Агента 2 много тяжёлых видео, он может не успеть к фиксированному часу, и
Агент 3 стартовал бы с неполными данными. Вместо этого:

1. Начиная с 08:00 (время старта Агента 1) Агент 3 периодически проверяет состояние общих очередей
   (`deep_parsing_agent.agent3_handoff_queue` и, в будущем, `intelligence_agent.search_results`) —
   без необходимости доработок в коде Агента 1/2, т.к. Агент 3 читает их напрямую.
2. Как только новых элементов не появляется **15 минут подряд** — пакет считается готовым, batch-анализ
   запускается.
3. **Защитный потолок — 11:00** (максимум 3 часа ожидания с 08:00): если очередь так и не
   "успокоилась" естественным образом, Агент 3 стартует принудительно с тем, что накопилось,
   помечает недостающее в `meta` результата и уведомляет пользователя, что часть данных не вошла в
   прогон.

Шаги внутри прогона (LangGraph-граф, §2 файла 4):

1. **Dispatcher** — собрать все накопленные с прошлого прогона job'ы, разбить на независимые куски
   работы (по одному на job или на кластер связанных job'ов).
2. **Параллельный анализ (`Send`)** — на каждый кусок: извлечь атомарные claims (`{subject,
   predicate, object, qualifiers}`) из `result`, сопоставить с уже известными `entities`.
3. **Reducer** — собрать все claims прогона в одном месте для кросс-сверки.
4. **Дедупликация** — кластеризация claims по embedding similarity (cosine ≥ **0.85**, значение по
   умолчанию, калибруется по факту) + LLM-judge подтверждение перед фактическим слиянием (не
   схлопывать по одной эвристике без проверки, §2.2 исследования).
5. **Детекция противоречий** — среди claims с одинаковым `(subject, predicate)`, но разным `object`:
   LLM-as-judge со структурированным выводом `{label: agree|contradict|unclear, confidence,
   explanation}`. Для случаев с высокой ценой ошибки (противоречие с claim высокой историчной
   уверенности) — self-consistency: **3 сэмпла + голосование большинством** вместо одного вызова
   судьи (§2.2 исследования).
6. **Контроль качества входа** — для каждого claim с низким `confidence.level` от Агента 2/1:
   применить логику эскалации (§2.3 выше).
7. **GlobalSynthesis** — финальный проход `claude-sonnet-4-6`: собрать дайджест (§3.2), посчитать
   количественные агрегаты (число источников, охват, динамика).
8. Сохранить результат прогона (дайджест + все claims + connections) в Supabase, пометить job'ы как
   обработанные.

### 2.3 Контроль качества и эскалация (уже решено в `1. Идея.md`)

- confidence `≤ порога` → повторный разбор автоматически (video-pipeline MCP или другой инструмент
  по типу контента), лимит — **1 автоповтор на job**.
- Ожидаемая стоимость автоповтора **> $0.10** ИЛИ уже был 1 автоповтор → **не повторять
  автоматически**, добавить в `pending_user_decisions`, отправить уведомление в Telegram (§4.3).
- Если суммарная трата на автоповторы за прогон достигает **$5** → остановить дальнейшие
  автоповторы в этом прогоне, уведомить пользователя, продолжить синтез с тем, что есть.

### 2.4 Дедупликация и противоречия — детали

- Порог similarity для кандидатов на дедуп — 0.85 (косинусное сходство эмбеддингов Gemini
  Embedding 2). Кандидаты **не** схлопываются автоматически без LLM-judge подтверждения.
- Противоречие с claim, у которого исторический confidence "высокий" (устоявшийся факт) —
  результат идёт не тихой заменой, а отдельным пунктом `contradictions` в дайджесте с explicit
  предупреждением (§1 брейншторма, "Алерт на противоречие").

### 2.5 Накопление истории (для v2, но структура нужна с MVP)

- Каждый claim хранит `extracted_at`, ссылку на `source_id`, и (если применимо) `superseded_by`
  (ссылка на claim, который его заменил/уточнил) — задел под будущий алгоритм отслеживания
  нарратива, ничего специально считать в MVP не нужно, только сохранять связи.
- Таблица `sources` уже с MVP хранит связь source → все его claims — задел под будущий рейтинг
  источников (v2), сам скор в MVP не считается.

## 3. Интерфейсы

### 3.1 Вход

**От Агента 2** (уже описан, `8. Архитектура.md` Агента 2 §7.2):
```json
{
  "job_id": "uuid",
  "content_type": "video | audio | text | document | image",
  "result": { "...": "..." },
  "confidence": { "level": "высокая|средняя|низкая", "explanation": "текст" },
  "meta": { "tools_used": [], "duration_sec": 0, "cost_usd": 0, "cache_hit": false }
}
```

**От Агента 1** (реализовано, таблица `intelligence_agent.search_results`, читать через Supabase
SDK по `telegram_id`, сортировка `created_at DESC` — точная схема подтверждена чтением исходников
Агента 1, `Code/src/orchestrator/index.js` + `Code/src/db/migrations/002_search_results.sql`):
```json
{
  "job_id": "uuid",
  "telegram_id": 0,
  "task_type": "report | trends | search",
  "query_topics": ["..."],
  "depth": "quick | standard | deep",
  "status": "ok | partial | error",
  "result": { "raw": { "perplexity": {}, "youtube": [], "firecrawl": [] } },
  "telegram_text": "готовый Telegram-отчёт — fallback, если result.raw пуст",
  "confidence": { "level": "высокая|средняя|низкая", "explanation": "текст" },
  "meta": { "tools_used": [], "duration_sec": 0, "cost_usd": 0, "model_used": "" },
  "created_at": "ISO-datetime"
}
```

### 3.2 Выход — MCP-инструменты Агента 3

**`analysis_digest`** — краткий дайджест последнего (или указанного) прогона:
```json
{
  "digest_id": "uuid",
  "run_at": "ISO-datetime",
  "facts": [
    {
      "claim_id": "uuid",
      "statement": "текст факта, сжатая форма конкретных данных",
      "confidence": {
        "level": "высокая|средняя|низкая",
        "sources_count": 3,
        "reach_estimate": 120000
      },
      "detail_ref": "claim_id — передать в analysis_detail за полной картиной"
    }
  ],
  "contradictions": [
    { "claim_a_id": "uuid", "claim_b_id": "uuid", "explanation": "текст" }
  ],
  "meta": {
    "items_processed": 0,
    "escalations_auto": 0,
    "escalations_pending_user": 0,
    "cost_usd": 0.0,
    "duration_sec": 0
  }
}
```

**`analysis_detail(claim_id)`** — полный разбор конкретного факта:
```json
{
  "claim_id": "uuid",
  "statement": "текст факта",
  "sources": [
    { "source_id": "uuid", "type": "video|audio|text|...", "ref": "job_id/url",
      "excerpt": "цитата/фрагмент, откуда взят факт", "confidence": "высокая" }
  ],
  "reasoning": "объяснение LLM-judge при слиянии/сверке, если применимо",
  "history": []
}
```

**`analysis_status`** — состояние текущего/последнего прогона:
```json
{
  "last_run_at": "ISO-datetime",
  "status": "ok | partial | error | cost_cap_reached",
  "items_processed": 0,
  "cost_usd": 0.0,
  "pending_user_decisions": [
    { "job_id": "uuid", "question": "текст", "estimated_cost_usd": 0.0 }
  ]
}
```

### 3.3 Уведомления (Telegram)

Прямой вызов Telegram Bot API (решение из брейншторма §3/§5) при:
- эскалации дороже $0.10 или повторной попытке — с объяснением проблемы и стоимостью повтора;
- достижении cost cap $5 за прогон;
- обнаружении противоречия с устоявшимся (высокий исторический confidence) фактом.

## 4. Модель данных (Supabase, схема `information_analysis_agent`)

| Таблица | Назначение |
|---|---|
| `entities` | Сущности (id, name, type, first_seen_at, last_seen_at) |
| `sources` | Источники (id, agent 1\|2, source_type, ref, raw_job_id) — задел под рейтинг v2 |
| `claims` | Факты (id, subject_entity_id, predicate, object_entity_id/value, confidence_level, confidence_explanation, source_id, extracted_at, superseded_by) |
| `runs` | История батч-прогонов (id, run_at, status, cost_usd, items_processed) |
| `pending_user_decisions` | Очередь эскалаций, ждущих ответа пользователя |
| pgvector-колонка на `claims`/`entities` | Эмбеддинги (Gemini Embedding 2) для similarity-дедупликации и семантического поиска |

## 5. Нефункциональные требования

- **Устойчивость** — LangGraph checkpoints, прогон не должен начинаться с нуля при сбое процесса.
- **Стоимость** — управляется на трёх уровнях: порог автоповтора $0.10/item, лимит 1 автоповтор на
  item, cost cap $5/прогон.
- **Наблюдаемость** — все LLM-вызовы через Helicone-прокси; трейсинг графа через LangSmith.
- **Единый язык стека** — Node.js (ESM) + `@langchain/langgraph`, консистентно с Агентом 1 и 2.

## 6. Явно не в MVP (v2)

- Алгоритм рейтинга источников (данные копятся, скор не считается).
- Алгоритм отслеживания нарратива во времени (данные копятся через `superseded_by`, анализ не
  считается).
- Адаптивная частота анализа (переход с ежедневного на реже, по предложению системы).
- Формальная статистическая калибровка confidence-порогов.
