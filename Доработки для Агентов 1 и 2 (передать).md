## ✅ Исправлено (2026-07-09) — конвейер не читал реальные данные Агента 1 и Агента 2

Это была не задача для Агента 1 или Агента 2 — баг был в коде **самого Агента 3**
(`Information analysis agent/Code/src/ingestion/agent1Reader.js` и `agent2Reader.js`). Оставляю
описание здесь первым пунктом для истории и потому что он объясняет и обесценивает часть того,
что раньше казалось уже проверенным (см. коммиты `304577c` — сам фикс, и `b5126f5` — сопутствующий
редеплой контейнера на `v0.6.0`). **Статус: исправлено, покрыто regression-тестами, подтверждено
живым запросом к проду.**

**Что обнаружено.** Supabase-клиент Агента 3 создаётся с фиксированной схемой:

```js
// Code/src/db/client.js
createClient(url, serviceKey, { db: { schema: 'information_analysis_agent' } })
```

Но `search_results` физически живёт в схеме `intelligence_agent` (Агент 1), а
`agent3_handoff_queue`/`parsing_results`/`parsing_jobs` — в схеме `deep_parsing_agent` (Агент 2).
Код читает их как `db.from('search_results')` / `db.from('agent3_handoff_queue')` — **без**
`.schema('intelligence_agent')` / `.schema('deep_parsing_agent')`. Я проверил это вживую прямо
сейчас реальным запросом через настоящий `@supabase/supabase-js` клиент с реальными продовыми
ключами:

- `db.from('search_results')` → `404 PGRST205: Could not find the table
  'information_analysis_agent.search_results' in the schema cache`
- `db.from('agent3_handoff_queue')` → та же ошибка, другая таблица
- `db.schema('intelligence_agent').from('search_results')` → **успех**, вернул реальную живую
  запись (прогон Агента 1 от 2026-07-07)
- `db.schema('deep_parsing_agent').from('agent3_handoff_queue')` → **успех**, вернул реальную
  запись (в очереди сейчас 13 pending job'ов от Агента 2, ждут Агента 3)

**Что это значит на практике.** `pollQueues.js` вызывает эти два ридера через `Promise.allSettled`
и при ошибке просто пишет `console.error` и продолжает с пустым результатом (специально, чтобы
сбой одного источника не ронял другой). Из-за этого **каждый плановый прогон Агента 3 молча
получает 0 новых элементов от Агента 1 и Агента 2**, даже когда данные реально есть (сейчас в базе
лежит минимум 1 необработанная запись Агента 1 и 13 — Агента 2). Ни одна ошибка при этом не всплывает
пользователю — просто "нечего анализировать". Это объясняет фразу в `Deep parsing agent/10.
Задачник.md`: «реального сквозного вызова ещё не было» — с точки зрения планового опроса очереди
это действительно так, несмотря на то что прямые вызовы (MCP-эскалация в Слайсе 7, живые
сквозные прогоны Слайсов 3/6/8) заводили тестовые данные другим путём и поэтому этой ошибки не
видели.

**Что сделано.** Добавлен `.schema('intelligence_agent')` в `agent1Reader.js` и
`.schema('deep_parsing_agent')` во все три запроса `agent2Reader.js`. При живой проверке фикса
вскрылся второй, ранее замаскированный первым баг: `agent2Reader.js` запрашивал
`agent3_handoff_queue.created_at`, а такой колонки в реальной таблице нет (проверено live-схемой:
`id/job_id/result_ref/attempt_count/last_attempt_at/status`) — `created_at` теперь берётся из
`parsing_jobs`, где он реально есть. `fakeSupabase.js` получил `db.schema(name)`, добавлены
regression-тесты, проверяющие `db.schemaCalls`. Живая проверка после фикса: `fetchAgent1Items`
вернул 1 реальный элемент, `fetchAgent2Items` — 5 (limit тестового вызова, в очереди 13),
`pollQueues` — 14 объединённых элементов с корректным `newestSeenAt`. 294/294 теста проходят.

## 🔴 Отдельно: обнаружена проблема безопасности (RLS отключён)

Побочный результат той же проверки — Supabase-advisor сообщил, что **у 12 таблиц из 3 схем
(`information_analysis_agent`, `intelligence_agent`, `deep_parsing_agent`) отключён Row Level
Security**, то есть все они полностью открыты на чтение/запись любому, у кого есть `anon`-ключ
проекта. Это не специфично для Агента 1/2 — касается всех трёх агентов сразу, поэтому не оформляю
как отдельные задачи для конкретного агента, а фиксирую здесь как общий пункт. Я **не применяю**
исправление сам — включение RLS без политик заблокирует весь текущий доступ сервисной роли, а
она используется во всех трёх агентах:

```sql
ALTER TABLE information_analysis_agent.claims ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.claim_sources ENABLE ROW LEVEL SECURITY;
ALTER TABLE intelligence_agent.reports ENABLE ROW LEVEL SECURITY;
ALTER TABLE intelligence_agent.search_results ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.runs ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.entities ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.digests ENABLE ROW LEVEL SECURITY;
ALTER TABLE intelligence_agent.api_costs ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.sources ENABLE ROW LEVEL SECURITY;
ALTER TABLE intelligence_agent.users ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.contradictions ENABLE ROW LEVEL SECURITY;
ALTER TABLE information_analysis_agent.pending_user_decisions ENABLE ROW LEVEL SECURITY;
```

Если вы захотите включить RLS, потребуются политики, разрешающие `service_role` полный доступ
(все три агента ходят через сервисный ключ, не через anon-сессии пользователей) — иначе всё
сломается. Это решение — на ваше усмотрение, не блокирует остальную работу.

## Раздел A — Агент 1 (Intelligence agent)

Ниже — только то, что **ещё не отмечено `[x]`** в `Intelligence agent/План разработки/06.
Задачник.md` и напрямую влияет на связку с Агентом 3. Не дублирую их собственный трекер — только
привязываю к контексту полного конвейера.

1. **Маршрутизация «сложного» контента к Агенту 2** (уже описано в их `06. Задачник.md`,
   раздел 1.5 «Новые задачи: переделка под модель «только поиск»» и раздел 1.6). Сейчас
   `agents/scout/index.js` использует Firecrawl только как поисковый инструмент
   (`/v1/search`, не полноценный парсинг страниц) — это соответствует модели «только поиск».
   Но логики «этот контент требует глубокого разбора → передать Агенту 2 через Redis/MCP» в коде
   ещё нет (пункты «Добавить логику передачи задачи Агенту 2» и «Принимать результаты от Агента 2»
   в их трекере не отмечены). Без этого шага сложный контент, который находит Агент 1
   (длинные видео, целые страницы), не попадает автоматически в очередь Агента 2 —
   должен закрываться отдельно командой Агента 1/2, не Агентом 3.

2. **Ручное тестирование записи в `search_results`** (их `06. Задачник.md`, раздел 1.11
   «Тестирование», 4 пункта не отмечены). Сама запись в код добавлена и подтверждена чтением
   исходников (см. уже существующий файл `Доработки для Агента 1 (передать).md` в этой же папке —
   статус «Реализовано»), но живого теста именно записи в БД со стороны Агента 1 не было.
   Раньше это было бы бессмысленно тестировать — Агент 3 не мог прочитать `search_results` из-за
   бага со схемой (см. пункт выше, уже исправлено 2026-07-09) — теперь путь открыт, можно
   тестировать по-настоящему.

3. **LangSmith/Helicone для Агента 1** (их раздел 1.4.1, помечено 🟡 важно). У Агента 3 оба уже
   подключены и проверены живыми вызовами (см. `9. Задачник.md` этого проекта, раздел
   «Наблюдаемость»); решение о том, чтобы настраивать одинаково для всех агентов разом, уже
   принято раньше (см. память `feedback_language_russian`-соседний пункт, у Агента 3 отмечено как
   «отложено — сделать для всех агентов разом»). Когда дойдёте — можно напрямую скопировать
   рабочую конфигурацию из `Code/.env.example` Агента 3 (`LANGCHAIN_TRACING_V2`,
   `LANGCHAIN_API_KEY` — использовать **Personal API key**, не Service API key: у Агента 3 Service

4. **Режим «Создание контента» (`mode_content`)** сейчас показывает заглушку «Агент 4 в
   разработке» (их раздел 1.12). Это не входит в объём текущей доработки — реальная маршрутизация

## Раздел B — Агент 2 (Deep parsing agent)

1. **Обновить `10. Задачник.md`, строку в разделе «Деплой»**: «Агент 3 не задеплоен (сам Агент 3
   ещё не задеплоен)» и «Агент 3 не задеплоен — сетевой путь и буферизация готовы, но реального
   сквозного вызова ещё не было» — оба утверждения устарели. Агент 3 задеплоен в Docker с версии
   v0.3.0 (Слайс 4), а реальный сквозной вызов MCP-эскалации к Агенту 2 подтверждён живым тестом в
   Слайсе 7 (`9. Задачник.md` этого проекта: «Docker Desktop + живые контейнеры Агента 2
   подняты... подтверждены ОБА сценария»). Единственное, что подтверждено **не было** — плановый
   опрос `agent3_handoff_queue` через `pollQueues` (см. критичный пункт выше) — стоит сформулировать
   именно так, а не «Агент 3 не задеплоен».

2. **`parsing_results` не хранит метаданные стоимости/инструментов.** У Агента 1
   (`intelligence_agent.search_results.meta`) есть `tools_used`/`cost_usd`/`duration_sec`/
   `model_used` — Агент 3 уже умеет это использовать (например, для `reach_estimate` из
   YouTube-метрик, Слайс 8). У Агента 2 в `parsing_results`/`parsing_jobs` таких полей нет
   (проверено чтением `Code/src/queue/index.js` — запись туда не включает `meta`). Это не
   блокирует текущую работу Агента 3 (он это уже учитывает как известное ограничение — см.
   комментарий в `agent2Reader.js`), но если в будущем понадобится взвешивать источники Агента 2
   по стоимости/инструментам так же, как источники Агента 1 — понадобится такое поле. Не срочно,
   но стоит знать.

3. **Схема данных — поправка.** При фиксе критичного пункта выше вскрылась ещё одна неточность
   (уже исправлена в Агенте 3, здесь — для сведения на будущее): `agent3_handoff_queue` реально
   состоит из `id, job_id, result_ref, attempt_count, last_attempt_at, status` — **колонки
   `created_at` в ней нет** (`agent2Reader.js` ошибочно её запрашивал). Метка времени берётся из
   `parsing_jobs.created_at`, которая там реально есть. Остальная структура —
   `parsing_results` (`job_id, module, result_json, confidence_level, confidence_text`),
   `parsing_jobs` (`content_type, content_ref, created_at`) — совпадает с ожиданиями
   `agent2Reader.js`. Дорабатывать здесь нечего, это уже поправлено на стороне Агента 3.
