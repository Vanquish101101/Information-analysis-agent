## 2. Контракт Агента 3 — что реально возвращает MCP-сервер

Агент 3 — read-only MCP-сервер, Streamable HTTP, порт **7302**, путь `/mcp`, контейнер
`information-analysis-agent`, сеть `marketing-agency-net`. Три инструмента (проверено чтением
`Code/src/mcp-server/server.js`/`queries.js`, не только ТЗ — код и документ совпадают):

### `analysis_digest({ run_id?: string })`

Дайджест последнего (или указанного) прогона:

```json
{
  "digest_id": "uuid",
  "run_at": "ISO-datetime",
  "facts": [
    {
      "claim_id": "uuid",
      "statement": "связное предложение, сформулированное LLM из subject/predicate/object",
      "confidence": {
        "level": "высокая|средняя|низкая",
        "sources_count": 3,
        "reach_estimate": 120000
      },
      "detail_ref": "тот же claim_id — передать в analysis_detail за полной картиной"
    }
  ],
  "contradictions": [
    { "claim_a_id": "uuid", "claim_b_id": "uuid", "explanation": "текст" }
  ],
  "meta": {
    "items_processed": 0,
    "escalations_auto": 0,
    "escalations_pending_user": 0,
    "cost_usd": 0.0,
    "duration_sec": null
  }
}
```

**Реальные нюансы, которых нет в ТЗ:**
- `meta.duration_sec` **всегда `null`** — узел `globalSynthesis.js` не имеет доступа к моменту
  старта прогона, это осознанно отложено (см. комментарий в коде: «задел под ТЗ §3.2, реальное
  значение появится, когда где-то в графе начнёт трекаться `startedAt`»). Не полагайтесь на это
  поле.
- `sources_count`/`reach_estimate` — настоящие, честные числа (RPC `claim_source_stats` по живым
  данным `claim_sources`/`sources.reach_estimate`), не заглушки. `reach_estimate` сейчас считается
  только для YouTube-результатов Агента 1 (`views + likes`), для остальных типов источников — 0.
- Если за прогон не сохранился дайджест (например, `persistedFacts` пуст) — вернётся пустая форма:
  `{ digest_id: null, run_at: null, facts: [], contradictions: [], meta: null }`, не ошибка.

### `analysis_detail({ claim_id: string })`

Полный разбор конкретного факта:

```json
{
  "claim_id": "uuid",
  "statement": "Subject: predicate: object_value",
  "sources": [
    { "source_id": "uuid", "type": "video|audio|text|search|...", "ref": "raw_job_id",
      "excerpt": null, "confidence": "высокая" }
  ],
  "reasoning": "confidence_explanation факта",
  "history": []
}
```

**Реальные нюансы:**
- `sources[].excerpt` — **всегда `null`**. ТЗ обещает «цитата/фрагмент, откуда взят факт», но
  `getClaimDetail` (`queries.js`) никогда его не заполняет. Если Агенту 4 для контента нужна
  реальная цитата/фрагмент исходника — этого поля недостаточно, нужно идти за сырыми данными (см.
  §3 ниже).
- `history` — **всегда пустой массив**, не реализовано вообще.
- `sources[].ref` = `raw_job_id` — это ключ, которым можно вернуться к исходной записи в
  `intelligence_agent.search_results` или `deep_parsing_agent.parsing_jobs`/`parsing_results` за
  сырыми данными (см. §3).
- Если `claim_id` не найден — не ошибка транспорта, а `{ isError: true, content: [...] }` с
  понятным текстом (MCP-конвенция), проверяйте `isError`, не полагайтесь на HTTP-статус.

### `analysis_status()`

```json
{
  "last_run_at": "ISO-datetime",
  "status": "ok | partial | error | cost_cap_reached",
  "items_processed": 0,
  "cost_usd": 0.0,
  "pending_user_decisions": [
    { "job_id": "uuid", "question": "текст", "estimated_cost_usd": 0.0 }
  ]
}
```
Полностью реальные данные, без заглушек — прямое чтение `runs`/`pending_user_decisions`.

## 3. Если синтезированных фактов Агента 3 недостаточно — сырые данные лежат отдельно

Агент 3 намеренно **не хранит** сырой исходный контент — только извлечённые
subject/predicate/object-факты. Если Агенту 4 для создания контента понадобится что-то более
богатое (полная цитата, полный транскрипт видео, полное описание кадра, полный текст статьи) —
через `analysis_detail.sources[].ref` (это `raw_job_id`) можно дойти до исходной записи:

- **От Агента 1** (`intelligence_agent.search_results`, `job_id` = `raw_job_id`): поле
  `result.raw` содержит `perplexity.summary`+`citations`, полный список YouTube-видео
  (`title/views/likes/url/channel/description`), полный список Firecrawl-страниц
  (`title/url/content`). Схема подтверждена и реально читается Агентом 3 (после фикса 2026-07-09,
  см. `Доработки для Агентов 1 и 2 (передать).md`).
- **От Агента 2** (`deep_parsing_agent.parsing_results`, `job_id` = `raw_job_id`): поле
  `result_json` содержит полный результат разбора модуля (полный транскрипт видео/аудио, полный
  OCR-текст, описание визуального ряда — зависит от модуля). Это данные **постоянно хранятся в
  Supabase**, не только эфемерно в Redis.

**Важно для реализации:** обе таблицы физически живут в схемах `intelligence_agent`/
`deep_parsing_agent`, а не в схеме, под которую будет пинован Supabase-клиент Агента 4. Смотрите
`.schema('intelligence_agent')`/`.schema('deep_parsing_agent')` — тот же баг, что был у Агента 3
(см. `Доработки для Агентов 1 и 2 (передать).md`, раздел про cross-schema чтение), легко повторить
заново, если не учесть сразу.

**Чего у Агента 2 нет нигде постоянно:** метаданные стоимости/инструментов (`cost_usd`/
`tools_used`/`duration_sec`) по конкретному parsing-job'у — считаются на лету, но не сохраняются в
Supabase (только эфемерно в Redis-кэше/буфере хендоффа, недоступны постфактум). Если для Агента 4
это важно — потребуется доработка на стороне Агента 2 (см. handoff-файл, раздел B, пункт 2).

## 4. Реальная топология и доступ

Все три работающих агента сейчас — контейнеры в одном локальном Docker Desktop на машине
пользователя, сеть `marketing-agency-net`, без публичного VPS ни у одного (см. handoff-файл,
раздел C, пункт 3 — «синхронная работа» сегодня означает «все контейнеры подняты одновременно
локально», не облачный прод). Порты: Агент 1 — Telegram-бот, без открытого HTTP наружу; Агент 2 —
MCP на 7301 (+ video-pipeline sidecar на 7310); Агент 3 — MCP на 7302. Общий Redis — 6379. Общая
Supabase — service_role-ключ у каждого агента в своём `.env`, RLS включён на всех таблицах, но
`service_role` его игнорирует (`BYPASSRLS`), так что доступ не изменится.

Для локальной разработки Агента 4: поднимать через тот же паттерн, что у остальных —
`docker-compose.yml` в своей папке + строка в мастер `Инфраструктура (Docker)/docker-compose.yml`
(сейчас там уже есть закомментированная заготовка строки под Агента 4).

проще/быстрее/качественнее, решится только при реальной реализации. Плюс отдельное видение:
интерактивное разрешение `pending_user_decisions` прямо в Telegram (сейчас Агент 3 только
уведомляет об эскалациях/противоречиях/cost cap, без возможности ответить). Оба видения **явно
отложены** пользователем — не начинать реализовывать без отдельного запроса. Подробности сохранены
в памяти Claude, файл `project_future_unified_bot_vision.md`.

`mode_content` в Агенте 1 существует как UI-заготовка (кнопка есть, состояние пока не сохраняется
даже частично — см. handoff-файл, раздел A, пункт 4) — это единственный кусочек кода, уже
