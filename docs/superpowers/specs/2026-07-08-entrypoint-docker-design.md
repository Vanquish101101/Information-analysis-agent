# Дизайн: Точка входа приложения + Docker-деплой (Агент 3)

**Дата:** 2026-07-08
**Статус:** Approved
**Соответствует:** `План разработки/8. Разработка.md` §1, §3, §4 (Шаг 1, Шаг 11), `План разработки/7. Архитектура (Бекенд).md` §3, §8.1

## Контекст

Планировщик батча (`createScheduler`) и граф анализа (`createAnalysisGraph`/`runAnalysis`) уже
готовы и протестированы по отдельности (81/81 тестов), но нигде не соединены — приложения,
запускающего их вместе, не существует. У Агента 3 также нет `Dockerfile`/`docker-compose.yml`, в
отличие от Агента 1 и Агента 2, которые уже задеплоены в контейнерах на общей сети
`marketing-agency-net`.

Ранее (при разработке планировщика) было решено отложить Redis и использовать
`InMemoryStateStore` — с явной оговоркой, что интерфейс `StateStore` спроектирован так, чтобы
подключение Redis позже не потребовало изменений в логике планировщика. Раз сейчас поднимается
Docker-деплой, решено подключить Redis в этом же слайсе (предельная стоимость сейчас ниже, чем
стоимость возврата к вопросу отдельным слайсом позже) — используя уже существующий общий
Redis-контейнер агентов 1/2 из общей инфраструктуры, а не поднимая новый.

## Решение

```
src/index.js (точка входа)
  ├── создаёт реальный Supabase-клиент (createSupabaseClient)
  ├── создаёт реальный LLM-экстрактор (createOpenRouterExtractor)
  ├── создаёт RedisStateStore (новый модуль, тот же интерфейс {get, set})
  ├── создаёт граф: createAnalysisGraph({db, extractClaims}) → runAnalysis
  ├── создаёт планировщик: createScheduler({db, stateStore, onBatchReady: runAnalysis, telegramId})
  └── scheduler.start(60_000) — проверка раз в минуту
```

## Компоненты

### 1. `src/scheduler/redisStateStore.js`

```js
createRedisStateStore({ redisUrl }) → { get(key): Promise<any|null>, set(key, value): Promise<void> }
```

Тот же интерфейс, что `createInMemoryStateStore()` — планировщик не знает и не должен знать, какая
реализация под капотом. Ключи с префиксом `scheduler:agent3:` (например,
`scheduler:agent3:watchStartedAt`) — без конфликта с ключами, которые Агент 1/2 могут писать в тот
же общий Redis. Значения — простые строки (ISO-даты, UTC-даты, `'true'`), TTL не требуется —
планировщик сам управляет актуальностью своего состояния (сбрасывает ключи при триггере).

Примечание: `get`/`set` в реальном Redis-клиенте асинхронны (`Promise`), в отличие от синхронного
`InMemoryStateStore`. Планировщик (`src/scheduler/index.js`) уже вызывает `stateStore.get`/`.set`
без `await` — это нужно проверить и поправить при реализации (`checkOnce()` должен `await`-ить
чтение/запись состояния независимо от реализации `StateStore`), либо явно задокументировать как
требование к интерфейсу `StateStore`: оба метода могут быть синхронными или возвращать `Promise`,
вызывающий код обязан их `await`-ить в обоих случаях.

### 2. `src/index.js` — точка входа приложения

Читает `.env` (`dotenv/config`), собирает зависимости через фабрики, запускает
`scheduler.start(60000)`. Логирует старт и необработанные ошибки в консоль (тот же стиль, что уже
используется в `persistResults.js`/`pollQueues.js` — `console.error` с префиксом модуля). Не
завершает процесс сам — контейнер держится за счёт `setInterval` планировщика (event loop не
опустошается).

### 3. `Code/Dockerfile`

Базовый образ той же версии Node.js LTS, что уже используют сиблинги (проверить точный тег при
реализации — Агент 1/2 используют конкретный тег, не просто `node:lts`, для воспроизводимости).
`npm ci --omit=dev` (без dev-зависимостей — их у Агента 3 нет, но паттерн на будущее), `CMD ["node", "src/index.js"]`.

### 4. `Code/docker-compose.yml`

Сервис `information-analysis-agent`, подключение к внешней сети `marketing-agency-net`
(`external: true`, тот же паттерн, что у `Deep parsing agent/Code/docker-compose.yml`),
`env_file: .env`, `restart: unless-stopped`. Никакого нового Redis-контейнера — подключение к уже
существующему общему через `REDIS_URL` из `.env` (значение уже используется Агентом 1:
`redis://marketing-agency-redis:6379`).

## Обработка ошибок

Ошибки на уровне отдельного тика планировщика уже изолированы (`checkOnce()` — try/catch,
возвращает `'WAITING'` при сбое). Ошибки на старте (например, недоступен Redis/Supabase при
запуске контейнера) — точка входа не должна пытаться "тихо" продолжать: если `createSupabaseClient`/
подключение к Redis падает при старте, процесс завершается с ненулевым кодом (Docker перезапустит
контейнер по `restart: unless-stopped`), а не работает в частично неисправном состоянии.

## Тестирование

`redisStateStore.js` — юнит-тест с инжектируемым фейковым Redis-клиентом (тот же DI-паттерн, что
везде в проекте) — не требует живого Redis для `npm test`. Интерфейс `{get, set}` идентичен
`InMemoryStateStore`, поэтому существующая логика планировщика (`decision.js`, `index.js`) и её
тесты не меняются вообще — переиспользуются как есть.

`src/index.js` — точка входа не юнит-тестируется по существу (как `start()/stop()` в планировщике
не тестировались вызовом реального `setInterval` дольше пары тиков) — проверяется вручную запуском
контейнера (`docker compose up`) и наблюдением логов после реализации.

## Явно не входит в этот слайс

- Изменение логики планировщика (`decision.js`, само поведение `checkOnce`) — только замена
  реализации `StateStore`.
- MCP-сервер, Telegram-уведомления, дедупликация, противоречия, эскалация, GlobalSynthesis —
  отдельные будущие слайсы.
- Подключение к главному деплой-файлу (`Marketing agency Project/Инфраструктура (Docker)/
  docker-compose.yml` через `include:`) — по конвенции проекта (см. `7. Архитектура (Бекенд).md`
  §3.3) делается отдельно, когда контейнер уже проверен через собственный `docker compose up`.

## Открытые вопросы

- Точный тег базового образа Node.js — сверить с `Dockerfile` Агента 1/2 при реализации, не
  угадывать заново.
- Библиотека Redis-клиента (`ioredis` vs `redis`) — проверить, какую уже использует общая
  инфраструктура/сиблинги, для консистентности; если нигде не используется — выбрать на этапе
  написания плана реализации, не архитектурное решение.
