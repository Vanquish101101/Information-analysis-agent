# Дизайн: Планировщик батча (Агент 3)

**Дата:** 2026-07-08
**Статус:** Approved (Вариант 1 — без Redis)
**Соответствует:** `План разработки/8. Разработка.md` §4 Шаг 3, `План разработки/7. Архитектура (Бекенд).md` §6.1, §8.1

## Контекст

Ingestion-слой (`agent1Reader.js`, `agent2Reader.js`, `normalizeItem.js`) уже готов и протестирован
без живой инфраструктуры. Следующий компонент по дорожной карте MVP — планировщик батча: он следит
за очередями Агента 1 (`intelligence_agent.search_results`) и Агента 2
(`deep_parsing_agent.agent3_handoff_queue`), определяет момент, когда батч готов к обработке, и
передаёт управление графу анализа LangGraph (граф ещё не построен — отдельный будущий слайс).

Архитектурный документ описывает состояние планировщика (курсор последнего увиденного элемента,
таймер тишины) как хранимое в Redis. У Агента 3 пока нет собственной инфраструктуры
(`docker-compose.yml`, `.env` с `REDIS_URL`, подключение к сети `marketing-agency-net`) — это ещё не
поднято. Redis как таковой уже существует в проекте как общая инфраструктура для Агентов 1/2, но
подключение к нему для Агента 3 добавило бы интеграционный (Level 2) тест-контур в задачу, которая
по сути про логику планировщика.

## Решение

Строим планировщик в два слайса:

- **Этот слайс (Вариант 1):** вся логика планировщика — чистые функции + `StateStore` как интерфейс,
  реализованный сейчас только как `InMemoryStateStore`. Полностью unit-тестируемо без Docker/Redis,
  без реального ожидания времени.
- **Следующий слайс (Вариант 2, отложено):** `RedisStateStore`, реализующий тот же интерфейс
  `StateStore`, плюс `docker-compose.yml`/`.env` для Агента 3. Логика планировщика (`decision.js`,
  `index.js`) при этом не меняется — меняется только то, какая реализация `StateStore` передаётся
  при создании планировщика. Разумное время для этого слайса — когда Агент 2 будет готов к
  соединению всех трёх агентов по архитектуре (пользователь отметил, что Агент 2 уже почти готов).

Эта граница (`StateStore`-интерфейс) выбрана специально, чтобы перенос на Redis не требовал
переписывания логики принятия решений и не мог внести туда баги — риск переноса ограничен одной
новой реализацией интерфейса из двух методов.

## Компоненты

### 1. `src/scheduler/decision.js`

Чистая функция без побочных эффектов:

```js
decideAction({ now, watchStartedAt, lastSeenAt, triggeredToday })
  → 'OUTSIDE_WINDOW' | 'WAITING' | 'BATCH_READY' | 'FORCED_CEILING'
```

- `OUTSIDE_WINDOW` — до начала окна наблюдения (`BATCH_MAX_WAIT_HOUR`-часовое окно стартует в 08:00
  по локальному времени) или батч на сегодня уже был запущен (`triggeredToday === true`).
- `WAITING` — окно наблюдения активно, но ни тишина 15 минут, ни потолок ещё не наступили.
- `BATCH_READY` — новых элементов не было `BATCH_IDLE_MINUTES` минут подряд (дефолт 15,
  из `.env`/дефолт в коде).
- `FORCED_CEILING` — наступил защитный потолок `BATCH_MAX_WAIT_HOUR` (дефолт 11:00), независимо от
  тишины в очереди.

Пороги читаются из `.env` (`BATCH_IDLE_MINUTES`, `BATCH_MAX_WAIT_HOUR`), с дефолтами в коде на случай
отсутствия переменных (15 и 11 соответственно, как уже зафиксировано в `.env.example` планов).

### 2. `src/scheduler/pollQueues.js`

Переиспользует существующие `fetchAgent1Items`/`fetchAgent2Items`. Принимает курсор
(`sinceTimestamp`), возвращает только элементы новее курсора и новую метку
`newestSeenAt` (максимум `created_at` среди полученных элементов, либо `null`, если новых элементов
нет).

```js
pollQueues(db, { telegramId, sinceTimestamp })
  → { items: NormalizedItem[], newestSeenAt: string | null }
```

Ошибки чтения одного источника (Агент 1 или Агент 2) не должны блокировать другой — если один из
двух вызовов падает, логируем и возвращаем то, что успешно получено от второго (частичный результат
лучше, чем полный отказ; оба источника уже независимо друг от друга опциональны по контракту
ingestion-слоя).

### 3. `src/scheduler/stateStore.js`

```js
createInMemoryStateStore() → { get(key), set(key, value) }
```

Простейшая обёртка над `Map`. Единственная реализация на этом слайсе. Ключи, которые планировщик
будет там хранить: `watchStartedAt`, `lastSeenAt`, `triggeredToday` — сериализуемые примитивы
(строки/булевы), чтобы будущий `RedisStateStore` мог хранить их как строки без изменений в
вызывающем коде.

### 4. `src/scheduler/index.js`

```js
createScheduler({ db, stateStore, onBatchReady, telegramId, now = () => new Date() })
  → { checkOnce(), start(intervalMs), stop() }
```

- `checkOnce()` — основная тестируемая единица одного тика:
  1. Прочитать состояние (`watchStartedAt`, `lastSeenAt`, `triggeredToday`) из `stateStore`.
  2. Если это первый тик за день (`watchStartedAt` не установлен и сейчас ≥ 08:00) — установить
     `watchStartedAt = now()`.
  3. `pollQueues()` — если пришли новые элементы, обновить `lastSeenAt` и накопить их в буфере
     батча (тоже в `stateStore`, как список id или сериализованных элементов — см. открытый вопрос
     ниже).
  4. Вызвать `decideAction()` с текущим состоянием.
  5. Если `BATCH_READY` или `FORCED_CEILING` — вызвать `onBatchReady(items, { reason })`, где
     `reason` — `'idle'` или `'ceiling'`; сбросить состояние (`triggeredToday = true`,
     `watchStartedAt = null`, `lastSeenAt = null`) для следующего дня.
  6. Если `WAITING`/`OUTSIDE_WINDOW` — ничего не делать, ждать следующего тика.
- `start(intervalMs)`/`stop()` — тонкая обёртка на `setInterval`/`clearInterval` вокруг `checkOnce()`
  для реального запуска процесса. Не является предметом юнит-тестирования по существу (только
  проверка, что `start` действительно планирует повторные вызовы, а `stop` их останавливает).

`onBatchReady` — единственная граница передачи управления дальше по цепочке. Сигнатура:
`onBatchReady(items: NormalizedItem[], { reason: 'idle' | 'ceiling' }) → void | Promise<void>`.
На этом слайсе граф анализа не существует — вызывающий код (или тест) передаёт заглушку
(например, логирующую функцию). Когда появится граф LangGraph, туда подставят реальный вызов графа
без изменений в `scheduler/index.js`.

## Данные vs транспорт (снятие возможной путаницы)

Сами данные батча идут строго через Supabase → память процесса → (будущий) граф → Supabase,
обычным вызовом функции внутри одного процесса — Redis в этом пути не участвует. Redis (в будущем
слайсе) хранит только служебное состояние планировщика для устойчивости к перезапуску процесса, и
отдельно — чекпоинты LangGraph для устойчивости графа к сбоям. Ни то, ни другое не является
"механизмом передачи данных" — это подстраховка на случай сбоя, ортогональная основному потоку
данных.

## Обработка ошибок

- Ошибка чтения одного источника в `pollQueues` — частичный результат, не блокирует тик (см. выше).
- Ошибка на любом другом шаге `checkOnce()` — логируется, тик считается как `WAITING` (следующий тик
  повторит попытку). Планировщик не должен падать полностью из-за одного неудачного тика.
- `onBatchReady`, выбросивший исключение — логируется, состояние всё равно сбрасывается на "новый
  день" (иначе планировщик будет бесконечно ретраить один и тот же батч каждые `intervalMs`).

## Тестирование

Полностью unit-уровень (Level 1 по `9. Тестирование.md`), без Docker/Redis:

- `decision.js` — таблица кейсов: до 08:00, в окне без тишины, тишина ровно 15 минут, тишина больше
  15 минут, ровно 11:00, после 11:00, уже `triggeredToday`.
- `pollQueues.js` — фейковый Supabase-клиент (`fakeSupabase.js`, уже есть), курсор корректно
  фильтрует старые элементы; частичный отказ одного источника не блокирует другой.
- `index.js` — фейковый db + `InMemoryStateStore` + управляемый `now()` + spy вместо
  `onBatchReady`; симуляция последовательности тиков в течение "дня" (подставляя разные `now`),
  проверка, что `onBatchReady` вызывается ровно один раз за день и с правильным `reason`.

## Явно не входит в этот слайс

- `RedisStateStore` и подключение к реальному Redis — следующий слайс (Вариант 2), после того как
  Агент 2 будет готов к полной интеграции трёх агентов.
- `docker-compose.yml`/`.env` для Агента 3 как отдельного контейнера — часть того же будущего слайса.
- Реальный вызов графа анализа внутри `onBatchReady` — граф ещё не существует, будет отдельным
  слайсом (Шаг 4 по `8. Разработка.md`).

## Открытые вопросы

- Буфер накопленных элементов батча между тиками: хранить целиком в `stateStore` (сериализовать
  список `NormalizedItem`) или просто расширять курсор и перечитывать весь диапазон одним запросом
  в момент `BATCH_READY`/`FORCED_CEILING`? Второй вариант проще и меньше риска рассинхронизации —
  предварительное решение: перечитывать диапазон одним запросом при триггере, не копить в
  `stateStore` построчно. Уточнить при написании плана реализации, если найдётся сложность.
