# Telegram-уведомления — дизайн (Слайс 10, ТЗ §3.3)

## 1. Назначение и границы

Автономные push-уведомления от Агента 3 пользователю в Telegram при трёх
проблемах, обнаруженных во время фонового батч-прогона (не связано с живым
диалогом пользователя — это отдельный сценарий, см. ниже):

1. Эскалация в `pending_user_decisions` (дороже $0.10 или неудачная попытка
   автоповтора).
2. Достигнут cost cap $5 за прогон.
3. Обнаружено противоречие с устоявшимся (исторически `confidence_level ===
   'высокая'`) фактом.

**Явно не в этом слайсе** (обсуждалось и намеренно отложено): единая система
маршрутизации живых запросов пользователя через Агента 1 → 2 → 3 → (4) —
отдельная, значительно более крупная архитектурная тема, затрагивающая
прежде всего Агента 1 (оркестратор), не этот проект. Слайс 9 (MCP-сервер)
уже даёт Агенту 1 интерфейс "потянуть" готовые результаты Агента 3 по
запросу — этого достаточно для будущей интеграции без работы в этом слайсе.

## 2. Источник токена

Переиспользуется `TELEGRAM_BOT_TOKEN` Агента 1 (общий бот на всю цепочку) —
не отдельная регистрация. `TELEGRAM_ALLOWED_USER_ID` у Агента 3 уже есть
(используется как фильтр при чтении очереди Агента 1) — тот же numeric id
используется как `chat_id` при отправке.

## 3. Реализация — без библиотеки

Агенту 1 нужен полноценный интерактивный бот (диалоги, команды) — там
`telegraf`. Агенту 3 нужна только простая исходящая отправка — обычный
`fetch` к `https://api.telegram.org/bot<token>/sendMessage`, тот же стиль,
что у всех остальных внешних вызовов в проекте (OpenRouter/Gemini — без
SDK-обёрток).

```javascript
// src/telegram/notify.js
export function createTelegramNotifier({ botToken, chatId, fetchImpl = fetch }) {
  // ...валидация botToken/chatId...
  return async function sendNotification(text) {
    const response = await fetchImpl(`https://api.telegram.org/bot${botToken}/sendMessage`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ chat_id: chatId, text, parse_mode: 'Markdown' })
    });
    // ...обработка ошибки, .json()...
  };
}
```

## 4. Место в графе

Новый узел `notifications` — последний узел графа, после `globalSynthesis`.
Не отправляет уведомления прямо из `escalation`/`contradiction` по ходу
прогона: если прогон позже всё равно упадёт с `status: error`, разосланные
раньше времени алерты о "проблемах" были бы вводящими в заблуждение —
собираем всё, что нужно знать, и отправляем одним пакетом только если
прогон реально завершился (`ok`/`partial`/`cost_cap_reached`).

Формат — **одно сводное сообщение на прогон**, разделами (эскалации / cost
cap / противоречия). Если проблем не было — сообщение не отправляется
вообще.

## 5. Данные для триггеров — что уже есть, чего не хватает

- **Эскалации**: `escalation.js` (Слайс 7) уже строит `{job_id, question,
  estimated_cost_usd}` на каждую строку `pending_user_decisions`, но
  никуда, кроме самой БД, это не отдаёт. Добавляется новый канал состояния
  `pendingDecisionMessages` — `escalation.js` возвращает те же объекты,
  что уже пишет в БД, узел `notifications` их просто отображает.
- **Cost cap**: `state.costCapReached` (Слайс 7) уже существует и
  переживает весь граф без изменений — узел `notifications` читает его
  напрямую, нового канала не нужно.
- **Противоречия**: `contradiction.js` (Слайс 6) уже читает
  `candidate.confidence_level` (историческая уверенность существовавшего
  факта) — но только чтобы решить, делать ли 3 замера вместо одного, и
  никуда это значение не сохраняет на возвращаемом claim'е. Добавляется
  новое поле `contradictedClaimHistoricalConfidence` на резолвленном
  claim'е (не новый top-level канал состояния — `state.claims` и так
  доходит без изменений до конца графа, ни `persistResults`, ни
  `globalSynthesis` его не перезаписывают). `notifications` фильтрует
  `state.claims` по `hasContradiction && contradictedClaimHistoricalConfidence
  === 'высокая'`.

## 6. Отказоустойчивость

Сбой отправки в Telegram (сеть, невалидный токен) логируется, не роняет
прогон — тот же принцип, что у каждого узла в этом графе.

## 7. Тестирование

Юнит-тесты `notify.js` (реальный паттерн `usage`-подобных LLM-файлов —
фейковый `fetchImpl`), юнит-тесты узла `notifications.js` (сборка текста
сообщения из всех трёх источников, пропуск отправки при отсутствии
проблем, отказоустойчивость на сбой отправки). Финальный живой смоук-тест
— реальное сообщение в реальный Telegram-чат при намеренно
спровоцированной проблеме (например, дешёвый тестовый прогон с
принудительным превышением порога эскалации).
